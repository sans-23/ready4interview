<p>
  In a monolithic architecture with a single database, keeping data consistent across multiple tables is easy: we start an ACID database transaction, perform updates, and either commit them all or roll back on failure.
</p>
<p>
  But in a distributed system where different microservices own separate databases, a single user request (e.g. buying a product) requires executing operations across multiple systems. Because we cannot execute a single database transaction across network boundaries, we need patterns to coordinate these multi-step processes.
</p>

<section className="content-section">
  <h2 id="core-problem">The Core Problem</h2>
  <p>
    Consider an e-commerce order process spanning three databases:
  </p>
  <ol>
    <li><strong>Order DB:</strong> Creates an order record in state `PENDING`.</li>
    <li><strong>Payment DB:</strong> Charges the user's wallet.</li>
    <li><strong>Inventory DB:</strong> Decrements item count.</li>
  </ol>
  <p>
    If the network drops, or the inventory service crashes after payment succeeds, our system is left in an inconsistent state: the user was charged, but the order is unfulfilled and stock was not allocated.
  </p>
  
  <Callout type="warning">
    <p>
      Avoid the "dual-write" anti-pattern: updating a local database and immediately publishing an event to a message broker in normal application code. If the database update succeeds but the server crashes before sending the message, downstream services will never know, leaving the system permanently out of sync.
    </p>
  </Callout>
</section>

  <p>
    <strong>How it works:</strong>
  </p>
  <ol>
    <li><strong>Phase 1 (Prepare):</strong> The coordinator sends a `Prepare` message to all participants. Each participant locks its local database resource, executes the transaction logic locally, and votes: `VOTE_COMMIT` (if successful) or `VOTE_ABORT` (if failed).</li>
    <li><strong>Phase 2 (Commit):</strong> If all participants voted `VOTE_COMMIT`, the coordinator broadcasts a `Commit` command. The participants write the updates and release their locks. If any participant voted `VOTE_ABORT` (or timed out), the coordinator broadcasts a `Rollback` command.</li>
  </ol>
  <p>
    <strong>Tradeoffs:</strong>
  </p>
  <ul>
    <li><strong>Pros:</strong> Guarantees strong consistency. The entire process behaves like a single transactional boundary (all-or-nothing).</li>
    <li><strong>Cons:</strong> **Highly Blocking**. Participants hold database resource locks between Phase 1 and Phase 2. If the coordinator crashes or the network lags during Phase 1, participants are stuck holding locks indefinitely, completely paralyzing the database. As a result, 2PC is rarely used in high-scale distributed microservices.</li>
  </ul>
</section>

<section className="content-section">
  <h2 id="saga-pattern">The Saga Pattern</h2>
  <p>
    The Saga pattern is the industry standard for coordinating multi-step workflows. Instead of holding global locks, a Saga is a sequence of **local database transactions**.
  </p>
  <p>
    Each service performs its local database update and emits an event. If a step fails, the Saga is responsible for executing **compensating transactions** in reverse order to rollback previous successful updates.
  </p>

  <Callout type="info">
    <p>
      A compensating transaction is a semantic undo. For example, if a step deducted $100 from a wallet, the compensating transaction is to credit $100. Sagas trade immediate consistency for eventual consistency: other transactions can read intermediate states while the Saga is running.
    </p>
  </Callout>

@Component
public class OrderSagaOrchestrator {
    
    @Autowired private PaymentClient paymentClient;
    @Autowired private InventoryClient inventoryClient;
    @Autowired private ShippingClient shippingClient;
    
    public void executeOrderWorkflow(Order order) {
        // Step 1: Charge Payment
        boolean paymentSuccess = paymentClient.charge(order.getUserId(), order.getAmount());
        if (!paymentSuccess) {
            updateOrderState(order.getId(), "FAILED_PAYMENT");
            return;
        }
        
        // Step 2: Reserve Inventory
        boolean inventorySuccess = inventoryClient.reserve(order.getProductId(), order.getQty());
        if (!inventorySuccess) {
            // Initiate rollback for Step 1
            paymentClient.refund(order.getUserId(), order.getAmount());
            updateOrderState(order.getId(), "FAILED_OUT_OF_STOCK");
            return;
        }
        
        // Step 3: Schedule Shipping
        boolean shippingSuccess = shippingClient.schedule(order.getId());
        if (!shippingSuccess) {
            // Initiate rollback for Step 2 and Step 1
            inventoryClient.release(order.getProductId(), order.getQty());
            paymentClient.refund(order.getUserId(), order.getAmount());
            updateOrderState(order.getId(), "FAILED_SHIPPING");
            return;
        }
        
        updateOrderState(order.getId(), "COMPLETED");
    }
}`} />

import logging

class OrderSagaOrchestrator:
    def __init__(self, payment_client, inventory_client, shipping_client):
        self.payment = payment_client
        self.inventory = inventory_client
        self.shipping = shipping_client

    def execute_saga(self, order_id: str, user_id: str, product_id: str, amount: float, qty: int) -> bool:
        # Define forward actions and their exact reverse compensation actions
        steps = [
            {
                "name": "charge_payment",
                "action": lambda: self.payment.charge(user_id, amount),
                "compensate": lambda: self.payment.refund(user_id, amount)
            },
            {
                "name": "reserve_stock",
                "action": lambda: self.inventory.reserve(product_id, qty),
                "compensate": lambda: self.inventory.release(product_id, qty)
            },
            {
                "name": "schedule_shipment",
                "action": lambda: self.shipping.schedule(order_id),
                "compensate": lambda: self.shipping.cancel(order_id)
            }
        ]

        executed_steps = []
        
        for step in steps:
            try:
                success = step["action"]()
                if not success:
                    raise RuntimeError(f"Step {step['name']} returned failure")
                executed_steps.append(step)
            except Exception as e:
                logging.error(f"Saga failed at {step['name']}. Triggering compensation...")
                self._rollback(executed_steps)
                return False
        return True

    def _rollback(self, executed_steps):
        # Compensations are executed in strict reverse order of execution
        for step in reversed(executed_steps):
            try:
                step["compensate"]()
            except Exception as e:
                # Log to dead-letter queue or persistent retry logic if compensation fails
                logging.critical(f"COMPENSATION FAILURE: {step['name']}. Requires manual intervention.")`} />

<section className="content-section">
  <h2 id="transactional-outbox">Transactional Outbox Pattern</h2>
  <p>
    In event-driven Sagas, services must update their local database and publish events to message brokers like Kafka. Doing this in two separate operations invites failures.
  </p>
  <p>
    The **Transactional Outbox Pattern** solves the dual-write problem by saving the event directly in the same database inside an `Outbox` table, using the same local transaction.
  </p>

    <strong>How it works:</strong>
  </p>
  <ol>
    <li>The application updates its entities (e.g. inserting an Order) and writes a message event row containing the payload to the `Outbox` table in the *same* database transaction.</li>
    <li>A separate background process—a **Message Relay** (which can be a Change Data Capture [CDC] engine like Debezium tailing the DB transaction log, or a simple database poller)—reads new rows from the `Outbox` table.</li>
    <li>The relay publishes the events to Kafka/RabbitMQ. Once published successfully, the relay deletes or marks the outbox rows as processed.</li>
  </ol>

    # Save Outbox Event in the exact same transaction
    outbox_event = Outbox(
        aggregate_type="ORDER",
        aggregate_id=new_order.id,
        payload=json.dumps(order_data)
    )
    session.add(outbox_event)
    
    session.commit()  # Commits atomically`} />
  </CodeTabs>

  <p>
    <strong>Tradeoffs:</strong>
  </p>
  <ul>
    <li><strong>Pros:</strong> Guarantees **at-least-once delivery** without distributed locks. Events are never lost if the server crashes immediately after committing business updates.</li>
    <li><strong>Cons:</strong> **At-least-once delivery means duplicates can occur**. The message relay might publish a message to Kafka, crash before marking the outbox row as processed, restart, and publish the same message again. Downstream consumers **must be idempotent**.</li>
  </ul>
</section>

<section className="content-section">
  <h2 id="problem-breakdowns">Problem Breakdowns (Multi-step in Practice)</h2>
  <p>
    Let's analyze how major system design problems apply these multi-step transaction patterns.
  </p>

  <h3>Ticketmaster / Flight Booking (Multi-Leg Reservations)</h3>
  <ul>
    <li><strong>Problem:</strong> A user books a multi-leg flight (NYC to London, London to Paris). If leg 2 booking fails, leg 1 must be cancelled.</li>
    <li><strong>Design Selection:</strong> **Orchestration-based Saga**. The flight reservation service acts as the Saga Orchestrator. It requests reservations for Leg 1 and Leg 2 sequentially. If Leg 2 fails (e.g. flight fully booked), the orchestrator triggers the compensation action to cancel Leg 1 and process refunds.</li>
  </ul>

  <h3>E-Commerce Order Fulfillment (Order Processing)</h3>
  <ul>
    <li><strong>Problem:</strong> The classic Payment &rarr; Inventory &rarr; Shipping workflow.</li>
    <li><strong>Design Selection:</strong> **Orchestration-based Saga + Transactional Outbox**. Using an Orchestrator makes the order lifecycle explicit and easy to audit. Combining it with the Transactional Outbox pattern guarantees that the Orchestrator's step transitions are published to Kafka reliably even under hardware crashes.</li>
  </ul>

  <h3>Cross-Bank Payments (Ledger Transfers)</h3>
  <ul>
    <li><strong>Problem:</strong> Transferring money from Bank A to Bank B. Failure to coordinate leaves the ledger unbalanced.</li>
    <li><strong>Design Selection:</strong> **Two-Phase Commit (2PC) or Choreography with Escrow**. If both banks run within a trusted, shared database cluster (e.g. Spanner), low-level 2PC guarantees atomic clearing. For separate banks, Bank A creates a pending debit and locks the funds in an "Escrow" state, emits an event, Bank B credits the target, and once Bank B sends an ACK event back, Bank A finalizes the debit.</li>
  </ul>
</section>

<section className="content-section">
  <h2 id="conclusion">Conclusion</h2>
  <p>
    Multi-step processes represent the shift from local databases to distributed systems. In interviews, establish whether the product requires strong immediate consistency (reaching for 2PC/Escrows) or eventually consistent Sagas, and always explain how you resolve the dual-write problem.
  </p>

  <div className="takeaway-card">
    <h3>
      <svg width="24" height="24" viewBox="0 0 24 24" fill="none" stroke="var(--primary)" strokeWidth="2"><path d="M12 2L2 7l10 5 10-5-10-5zM2 17l10 5 10-5M2 12l10 5 10-5"/></svg>
      Key Interview Takeaways
    </h3>
    <ul>
      <li><strong>Avoid 2PC for high scale:</strong> Mention 2PC to show theoretical knowledge of distributed ACID, but quickly rule it out for microservices due to blocking lock footprints.</li>
      <li><strong>Use Orchestrators for complex business flows:</strong> Centralized Sagas are much easier to trace and monitor in production compared to Choreography event spaghetti.</li>
      <li><strong>Combine Sagas with Outbox:</strong> Explain that Sagas publish events using the Transactional Outbox pattern so crashes do not result in lost messages.</li>
      <li><strong>Enforce consumer idempotency:</strong> Because outbox message relays guarantee at-least-once delivery, always state that consumers check event IDs against a processed database index.</li>
    </ul>
