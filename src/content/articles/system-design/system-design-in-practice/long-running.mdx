<section className="content-section">
  <h2 id="core-problem">The Core Problem: Thread Blocking</h2>
  <p>
    When a web server receives an HTTP request, it allocates a thread from its worker pool to handle it. If that thread executes a task that takes 30 seconds to complete, that thread is blocked.
  </p>
  <p>
    Executing heavy tasks inside the HTTP request loop results in three primary failures:
  </p>
  <ul>
    <li><strong>Gateway Timeouts (HTTP 504):</strong> Intermediaries (such as load balancers, CDN edges, or reverse proxies like Nginx) enforce maximum request timeouts—typically **30 to 60 seconds**. If the app server does not respond before this limit, the gateway terminates the connection, returning an error to the user even if the background server continues to churn.</li>
    <li><strong>HTTP Worker Thread Starvation:</strong> Most web servers operate with a finite number of connection threads (e.g. 200). If concurrent users trigger 200 long-running jobs, all threads are blocked. The server can no longer accept any new, fast requests (like serving a static page or checking auth), causing a complete outage.</li>
    <li><strong>Memory & Resource Exhaustion:</strong> Long-running tasks often manipulate massive arrays in memory (like reading 100,000 DB rows to generate a CSV). Concurrently running multiple copies of these processes can exhaust server RAM, triggering Out-Of-Memory (OOM) operating system crashes.</li>
  </ul>
</section>

  <h3>2. Server Push (WebSockets / Server-Sent Events)</h3>
  <p>
    Instead of clients polling, the server pushes completion updates in real-time. When a worker completes a task, it publishes an event (e.g. via Redis Pub/Sub) containing the `jobId` and output. 
  </p>
  <p>
    An active connection handler server (WebSockets/SSE gateway) subscribes to the channel and pushes the message down the persistent TCP socket to the client.
  </p>
  <p>
    <em>Tradeoff:</em> Drastically reduces server request rates and provides instantaneous UI updates, but introduces stateful connection management complexity at the API gateway layer.
  </p>
</section>

<section className="content-section">
  <h2 id="reliability-patterns">Reliability: Retries, DLQs & Idempotency</h2>
  <p>
    Decoupled architectures must survive worker crashes, networking glitches, and malformed task inputs. Ensure your system designs incorporate these reliability patterns:
  </p>

  <h3>1. Retry Policies (Exponential Backoff & Jitter)</h3>
  <p>
    If a worker fails to contact a third-party API or database during execution, the task should not fail immediately. Queues should support auto-retry. 
    To avoid slamming dependency systems, retries should use **Exponential Backoff** (delaying each retry longer than the last, e.g. 2s, 4s, 8s, 16s) combined with **Jitter** (adding a random noise offset, e.g. +/- 1.2s) to prevent coordinated "thundering herd" worker retry storms.
  </p>

  <h3>2. Dead Letter Queues (DLQs)</h3>
  <p>
    If a task contains invalid user variables, it will crash every worker that attempts to execute it. This is a **Poison Pill**. If left in the queue, it will consume worker resources and trigger infinite retry loops, locking up the system.
  </p>
  <p>
    A **Dead Letter Queue (DLQ)** acts as a quarantine. When a task fails more than a configured limit (e.g., 3 or 5 times), the broker pulls it from the active queue and moves it to the DLQ. Admins can inspect the malformed message payloads later without disrupting operational queues.
  </p>

  <h3>3. Task Idempotency</h3>
  <p>
    Most modern messaging brokers guarantee **At-Least-Once Delivery**. Due to network failures (such as a worker executing a task but crashing just before sending an acknowledgment back to the broker), the broker will re-deliver the message, causing it to run twice.
  </p>
  <p>
    To prevent duplicate actions (like charging a credit card twice), workers must execute tasks **idempotently**:
  </p>
  <ul>
    <li>The client includes an <code>idempotency_key</code> (such as a unique UUID) in the request payload.</li>
    <li>When the worker ingests the task, it checks if a record with the key exists in the database.</li>
    <li>Using a database constraint or Redis lock, the worker ensures that the operation is only executed if it hasn't run before. If a duplicate arrives, the worker simply skips execution and returns the cached result.</li>
  </ul>
</section>

<section className="content-section">
  <h2 id="problem-breakdowns">Problem Breakdowns (Long Tasks in Practice)</h2>
  <p>
    Let's analyze how major production environments optimize long-running task infrastructure.
  </p>

  <h3>YouTube / TikTok (Video Encoding Pipelines)</h3>
  <ul>
    <li><strong>Problem:</strong> Ingesting massive raw video uploads and converting them into 5+ HLS resolution formats, which demands high CPU usage.</li>
    <li><strong>Design Selection:</strong> **DAG Orchestration Workers + Priority Queues**. Videos are not transcoded by a single worker thread. Instead, they are parsed as Directed Acyclic Graphs (DAGs) and split into parallel sub-tasks (e.g. transcode segment 1, transcode segment 2). Dynamic scaling pools (auto-scaled by message queue depth) spin up computing nodes. Tasks are prioritized using separate broker queues: paid/premium account uploads go to high-priority queues, ensuring faster processing than free uploads.</li>
  </ul>

  <h3>SaaS Platforms (Heavy Report Exports)</h3>
  <ul>
    <li><strong>Problem:</strong> A user clicks "Export all data as CSV" in an analytics dashboard. Processing the database scan spikes CPU and locks tables.</li>
    <li><strong>Design Selection:</strong> **Rate-limited Batch Pulling + S3 Presigned Downloads**. Workers query the database in small batch chunks (e.g., 1,000 rows at a time using keyset pagination) rather than loading all rows into RAM. The worker streams the CSV results directly to S3. Once completed, the worker writes the S3 URL to the database and flags the job status as `COMPLETED`. The client UI, which was polling, receives the download URL and prompts the user.</li>
  </ul>

  <h3>Password Resets & transactional alerts</h3>
  <ul>
    <li><strong>Problem:</strong> Email APIs can experience transient timeouts, but users expect reset tokens within seconds.</li>
    <li><strong>Design Selection:</strong> **RabbitMQ Pub/Sub + DLQ**. The user registration/security controller writes the email payload to RabbitMQ and returns `200 Success` instantly. Independent mail worker threads pull from RabbitMQ and call the third-party provider (SendGrid/Mailgun). If the API is down, RabbitMQ retries using exponential backoff. If it fails repeatedly, the token payload goes to a DLQ so admins can monitor email delivery bottlenecks without locking the login screen.</li>
  </ul>
</section>

  <p>
    Keep your web servers lightweight. Offload intensive computations to background worker fleets, coordinate task queues using durable brokers, track statuses asynchronously, and enforce strict retries and idempotency controls.
  </p>

      Key Interview Takeaways

    <ul>
      <li><strong>Defend the thread pool:</strong> Always explain how running heavy operations on web threads leads to gateway timeouts and thread pool starvation.</li>
      <li><strong>Decouple with 202 Accepted:</strong> Return a job UUID immediately to keep client connections short and stateless.</li>
      <li><strong>Isolate bad tasks with DLQs:</strong> Discuss poison pills and how dead letter queues prevent bad input payloads from breaking your background worker pools.</li>
      <li><strong>Prevent double processing:</strong> Always mention idempotency keys checked at the worker layer to prevent duplicate task execution from at-least-once delivery queues.</li>
