<p>
  In system design interviews, most communication is assumed to be client-initiated request-response (pull-based). But when you need to design features like instant messaging, live bid updates, collaborative editing, or stock ticket streaming, the standard HTTP request-response cycle breaks down.
</p>
<p>
  This guide covers how to push updates from the server to the client, how to scale stateful connection layers to millions of concurrent sockets, and how to choose the right protocol based on technical tradeoffs and product requirements.
</p>

<section className="content-section">
  <h2 id="core-problem">The Core Problem</h2>
  <p>
    Standard HTTP/1.1 is **stateless and half-duplex**. A client sends an HTTP request, and the server returns an HTTP response. The server cannot initiate communication, nor can it send data to the client without a corresponding request.
  </p>
  <p>
    When a system requires real-time data flow (latency &lt; 500ms), engineers are faced with a fundamental choice: **Should the client constantly pull the server for updates, or should the server hold a persistent connection to push updates down?**
  </p>
  
  <Callout type="warning">
    <p>
      Many candidates default to saying "we will use WebSockets" for any real-time requirement. This is a common red flag. WebSockets introduce stateful servers, complex load balancing, and connection maintenance overhead. Always evaluate simpler options (like Short Polling or SSE) before jumping to WebSockets.
    </p>
  </Callout>
</section>

<section className="content-section">
  <h2 id="polling-patterns">Polling Patterns</h2>
  <p>
    Polling patterns shift the burden of real-time detection to the client. The client repeatedly queries the server to ask: <em>"Is there new data yet?"</em>
  </p>

  <h3 id="short-polling">Short Polling</h3>
  <p>
    With short polling, the client makes an HTTP request to the server at a fixed, short interval (e.g., every 2 or 5 seconds). The server immediately responds with either new data or an empty response if nothing has changed.

  <h3 id="long-polling">Long Polling</h3>
  <p>
    Long polling is an optimization over short polling. The client requests data, but instead of responding immediately with an empty payload, the server **holds the request open** (keeps the connection pending) until new data becomes available or a timeout occurs.
  </p>
  <p>
    Once new data is sent, the client immediately initiates a new long-polling request, starting the cycle again.
  </p>

<section className="content-section">
  <h2 id="push-protocols">Server-Push Protocols</h2>
  <p>
    Instead of polling, push protocols establish a single, long-lived connection between the client and server. The server can push data to the client instantly the moment it occurs.
  </p>

  <h3 id="sse">Server-Sent Events (SSE)</h3>
  <p>
    SSE is a standard HTML5 protocol built on top of persistent HTTP connections. The client opens an HTTP connection with the header `Accept: text/event-stream`. The server keeps this connection open and pushes events as a stream of plaintext lines.
  </p>

  <CodeBlock language="python">{`# Python FastAPI SSE Server Endpoint
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import asyncio
import json

app = FastAPI()

async def event_generator():
    # Push an event immediately upon connection
    yield "data: Connected to live events!\\n\\n"
    
    while True:
        await asyncio.sleep(2)
        # Yield data format: "data: {JSON}\\n\\n"
        data = json.dumps({"tick": int(asyncio.get_event_loop().time())})
        yield f"data: {data}\\n\\n"

@app.get("/stream")
async def stream():
    return StreamingResponse(event_generator(), media_type="text/event-stream")`}</CodeBlock>

  <p>
    <strong>Tradeoffs:</strong>
  </p>
  <ul>
    <li><strong>Pros:</strong> Native HTTP protocol. It automatically handles reconnection via the browser's `EventSource` API, includes built-in event IDs for message catching, and bypasses firewall issues because it runs over standard port 80/443. Crucially, when used with HTTP/2, multiplexing allows multiple SSE streams to share a single TCP connection.</li>
    <li><strong>Cons:</strong> Unidirectional. Data can only flow from **server to client**. If the client needs to talk back, it must initiate separate HTTP POST requests. It is also historically subject to a browser limit of 6 open connections per domain when running over HTTP/1.1 (though this limit is bypassed in HTTP/2).</li>
  </ul>

  <h3 id="websockets">WebSockets</h3>
  <p>
    WebSockets provide full-duplex, bidirectional communication over a single TCP connection. The connection starts as a standard HTTP request. The client requests a protocol change by sending a connection upgrade header (`Upgrade: websocket`). If the server agrees, it returns an HTTP `101 Switching Protocols` response, and the TCP socket remains open as a raw, bidirectional frame transport (`ws://` or `wss://`).
  </p>

  <CodeBlock language="python">{`# Python FastAPI WebSocket Endpoint
from fastapi import FastAPI, WebSocket, WebSocketDisconnect

app = FastAPI()

@app.websocket("/chat")
async def websocket_endpoint(websocket: WebSocket):
    # Accept client handshake and switch protocols
    await websocket.accept()
    try:
        await websocket.send_json({"type": "status", "message": "Connected!"})
        while True:
            # Receive incoming text/JSON frames
            data = await websocket.receive_json()
            # Process or dispatch message
            await websocket.send_json({"type": "echo", "payload": data})
    except WebSocketDisconnect:
        # Clean up connection maps on close
        print("Client disconnected")`}</CodeBlock>

  <p>
    <strong>Tradeoffs:</strong>
  </p>
  <ul>
    <li><strong>Pros:</strong> True bidirectional full-duplex communication. Low overhead per frame (just 2 to 14 bytes compared to kilobytes of HTTP headers), making it optimal for extremely high-frequency data streams (e.g., online multiplayer games, real-time whiteboards).</li>
    <li><strong>Cons:</strong> Stateful and complex. It does not use HTTP anymore, meaning you cannot rely on standard HTTP features like edge caching or compression out of the box. You must implement custom heartbeat ping-pongs to detect dead sockets, handle reconnection storms, and design complex load balancing strategies.</li>
  </ul>

  <h3 id="webrtc">WebRTC (P2P)</h3>
  <p>
    WebRTC (Web Real-Time Communication) is an open-source standard designed for direct **peer-to-peer (P2P)** communication between browsers. It allows audio, video, and raw data to flow directly between devices without going through a central server.
  </p>
  <p>
    <strong>Tradeoffs:</strong>
  </p>
  <ul>
    <li><strong>Pros:</strong> Ultra-low latency because packets bypass intermediary server hops. Ideal for real-time video/audio streaming and peer-to-peer file sharing. Runs over UDP.</li>
    <li><strong>Cons:</strong> High setup complexity. Requires signaling servers to negotiate connections, STUN/TURN servers to traverse NATs and firewalls, and does not scale well to large multicast audiences (e.g., streaming a live event to 10,000 viewers) because of peer bandwidth constraints.</li>
  </ul>
</section>

<section className="content-section">
  <h2 id="scaling-real-time">Scaling Real-time Systems</h2>
  <p>
    Scaling stateless application servers is easy: you simply spin up more instances behind a round-robin load balancer. However, scaling persistent real-time connections (WebSockets/SSE) introduces architectural bottlenecks.
  </p>

  </p>

  <p>
    The primary scaling bottlenecks are:
  </p>
  <ul>
    <li><strong>File Descriptors:</strong> Each open connection requires a file descriptor on Linux. The default OS limits (often 1024) must be adjusted (via `ulimit -n`) to support hundreds of thousands of concurrent connections.</li>
    <li><strong>Memory Footprint:</strong> Sockets require memory buffers. In Node.js, each socket can consume 10KB–30KB. Hosting 100,000 connections on a single machine requires several gigabytes of RAM dedicated solely to connection maintenance.</li>
    <li><strong>Load Balancing:</strong> Traditional round-robin load balancers can distribute initial HTTP requests evenly, but if 10,000 clients connect at the same time and hit the same load balancer node, it can create a localized spike. Load balancers must support WebSocket upgrades and route traffic using sticky routing or IP hashing.</li>
  </ul>

  <p>
    <strong>How it works:</strong>
  </p>
  <ol>
    <li>Every WebSocket server subscribes to a set of channels in Redis corresponding to the active rooms or users it currently hosts.</li>
    <li>When Client A sends a message to Client B via WS Server 1, Server 1 parses the message and publishes it to Redis (e.g., `PUBLISH chat_room_123 {...}`).</li>
    <li>Redis broadcasts the message to all WebSocket servers that have subscribed to `chat_room_123`.</li>
    <li>WS Server 2 receives the message from Redis, identifies that Client B has an active socket on its machine, and pushes the payload directly to Client B.</li>

import websockets
import redis.asyncio as aioredis
import json

# Local map tracking connections hosted on this physical server: userId -> WebSocket
local_connections = {}

async def redis_broadcast_listener(redis_uri: str):
    redis_client = aioredis.from_url(redis_uri)
    pubsub = redis_client.pubsub()
    await pubsub.subscribe("realtime_events")
    
    try:
        async for message in pubsub.listen():
            if message["type"] == "message":
                payload = json.loads(message["data"])
                target_user = payload.get("target_id")
                # Route the message if the user is connected to THIS machine
                ws = local_connections.get(target_user)
                if ws:
                    await ws.send(json.dumps(payload))
    except Exception as e:
        print(f"Redis Subscription error: {e}")

async def ws_handler(websocket, path):
    # Retrieve user credentials passed during the socket handshake
    user_id = websocket.request_headers.get("User-Id")
    local_connections[user_id] = websocket
    redis_client = aioredis.from_url("redis://localhost")
    
    try:
        async for message in websocket:
            data = json.loads(message)
            # Publish event to the Redis bus for distributed server consumption
            await redis_client.publish("realtime_events", json.dumps(data))
    except websockets.ConnectionClosed:
        pass
    finally:
        local_connections.pop(user_id, None)

async def main():
    uri = "redis://localhost"
    # Run the background listener to consume Redis pub/sub broadcasts
    asyncio.create_task(redis_broadcast_listener(uri))
    
    async with websockets.serve(ws_handler, "0.0.0.0", 8080):
        await asyncio.Future()  # run forever

if __name__ == "__main__":

<section className="content-section">
  <h2 id="failure-handling">Failure Handling & Resiliency</h2>
  <p>
    Unlike stateless REST services, connection failures in stateful systems have immediate downstream effects. A strong candidate will address the following resiliency strategies in their design.
  </p>

  <h3>Reconnection Storms (Thundering Herd)</h3>
  <p>
    If a WebSocket server hosting 500,000 connections crashes, all 500,000 clients will immediately attempt to reconnect. If they reconnect at the exact same moment, the incoming traffic spike will overwhelm the load balancers, firewalls, and application servers.
  </p>
  <p>
    <strong>Mitigation: Exponential Backoff with Jitter</strong><br/>
    Clients must not reconnect instantly. They must wait a randomized amount of time (jitter) that increases exponentially with each failed attempt.
  </p>

  <CodeBlock language="python">{`# Exponential backoff with jitter in Python
import random
import time

def get_reconnect_delay(attempt: int) -> float:
    max_delay = 30.0  # 30 seconds max
    base_delay = min(max_delay, 1.0 * (2 ** attempt))
    jitter = random.uniform(0, 1.0)  # 0-1 sec randomized buffer
    return base_delay + jitter`}</CodeBlock>

  <h3>Heartbeats (Ping/Pong)</h3>
  <p>
    Firewalls, load balancers, and routers often terminate connections that remain idle for too long. Additionally, if a client device suddenly loses battery or signal, the server may not receive a close frame, leaving a **"half-open connection"** that leaks memory.
  </p>
  <p>
    <strong>Mitigation:</strong><br/>
    The server regularly sends low-overhead `Ping` frames to the client. The client must respond with a `Pong` frame. If the server does not receive a response within a timeout window, it terminates the socket and cleans up local memory.
  </p>

  <h3>Message Delivery Guarantees</h3>
  <p>
    When a user is offline or reconnecting, messages sent to them can be lost. To prevent loss:
  </p>
  <ol>
    <li>**Sequence Numbers / Offsets:** Every message is assigned a monotonically increasing ID.</li>
    <li>**Client Synchronization:** Upon reconnecting, the client sends its last successfully processed ID (e.g., `last_received_id = 452`).</li>
    <li>**Replay Buffer:** The server queries the database or a cache for any events with `id > 452` and streams the missing events to sync the client.</li>
  </ol>

<section className="content-section">
  <h2 id="problem-breakdowns">Problem Breakdowns (Real-time in Practice)</h2>
  <p>
    Let's analyze how major system design problems apply these real-time principles.
  </p>

  <h3>Ticketmaster (Seat Map Updates)</h3>
  <ul>
    <li><strong>Requirement:</strong> Users browsing a concert seating layout need to see when seats are selected, reserved, or purchased by other buyers to avoid double-selection conflicts.</li>
    <li><strong>Design Selection:</strong> **Server-Sent Events (SSE)**. The data flow is unidirectional (updates flow from the server database to the browsers). The client does not need to send updates back over the same channel (they do so via normal REST actions like `POST /bookings`). Because SSE handles automatic reconnection and runs natively over HTTP/2, it is the perfect fit.</li>
  </ul>

  <h3>Uber (Driver Location Updates)</h3>
  <ul>
    <li><strong>Requirement:</strong> Ride-hailing passengers must see the driver's car moving smoothly on their map in real-time.</li>
    <li><strong>Design Selection:</strong> **WebSockets** or **Optimized HTTP Long Polling**. Drivers push their GPS coordinates every 3-4 seconds. Passengers receive these locations. Because the driver is constantly pushing updates and passengers are pulling them, WebSockets provides the bidirectional channel needed to handle the high frequency of client writes and server broadcasts.</li>
  </ul>

  <h3>WhatsApp (Chat Messaging)</h3>
  <ul>
    <li><strong>Requirement:</strong> Instant text messaging, typing indicators, read receipts, and online status.</li>
    <li><strong>Design Selection:</strong> **WebSockets**. Chat is highly bidirectional, low latency, and highly interactive. The client sends a packet, the server routes it, and the recipient acknowledges it. Because of the low framing overhead and true full-duplex nature, WebSockets is the industry standard.</li>
  </ul>

  <h3>Robinhood (Stock Price Streaming)</h3>
  <ul>
    <li><strong>Requirement:</strong> Users looking at a stock chart need to see price ticks update sub-second.</li>
    <li><strong>Design Selection:</strong> **SSE** or **WebSockets**. Because price feeds are strictly server-to-client broadcasts (the user does not write stock ticks back), SSE is a lightweight, easy-to-scale alternative. However, if the client is also running algorithmic trading tickers that need to stream orders back over the same low-latency connection, WebSockets is preferred.</li>
  </ul>

  <h3>Google Docs (Collaborative Editing)</h3>
  <ul>
    <li><strong>Requirement:</strong> Multiple users typing on the same document simultaneously, seeing cursor positions and character insertions update in real-time.</li>
    <li><strong>Design Selection:</strong> **WebSockets**. Edits require full-duplex communication. Clients send operations (Operational Transformations or CRDT states) and immediately receive edits from other collaborators. The bidirectional nature of WebSockets is critical to preventing editing conflicts.</li>
  </ul>

  <h3>Strava (Activity Syncing) — <em>Anti-Pattern Warning</em></h3>
  <ul>
    <li><strong>Requirement:</strong> Syncing a runner's path and run duration with their online profile.</li>
    <li><strong>Design Selection:</strong> **No Real-time Update Needed**. Running with a live socket drains mobile batteries, is highly unstable due to changing cell towers, and is unnecessary. Activity syncing must be done via **local storage caching** (SQLite/IndexedDB on device) and then batched over normal HTTP POST request retries once the run is completed or network stability is restored.</li>
  </ul>

  <h3>Online Auction (Live Bidding)</h3>
  <ul>
    <li><strong>Requirement:</strong> Users viewing a listing need to see the latest bid instantly. Bid latency must be sub-second to prevent disputes in the final seconds of an auction.</li>
    <li><strong>Design Selection:</strong> **WebSockets**. Bidding requires immediate processing, distributed lock check on the server, and a broadcast to all viewers. WebSockets ensures that both the bid submission and the immediate push to other clients bypass HTTP connection negotiation overhead.</li>
  </ul>

  <h3>FB Live Comments (Mass Broadcast Comments)</h3>
  <ul>
    <li><strong>Requirement:</strong> Millions of concurrent viewers watching a live stream, with comments flowing continuously down their screens.</li>
    <li><strong>Design Selection:</strong> **SSE Gateway + Redis Pub/Sub**. Running WebSockets for millions of viewers creates connection maintenance bottlenecks. Because comments are unidirectional (pushed to viewers), SSE is preferred. Scaled via a pool of lightweight event routers that pull from a Redis Pub/Sub cluster to broadcast comment events to edge viewers.</li>
  </ul>

  <h3>ChatGPT (Response Streaming)</h3>
  <ul>
    <li><strong>Requirement:</strong> Streaming words/tokens to the user's browser as the AI model generates them.</li>
    <li><strong>Design Selection:</strong> **Server-Sent Events (SSE)**. ChatGPT uses SSE because it handles token-by-token text streaming natively. The connection is unidirectional (only the server pushes text), and it works over standard HTTP/2, making it simple to manage and scale.</li>
  </ul>

  <p>
    Real-time updates are about choosing the right balance between complexity and latency. Always justify your choices by weighing connection state management against the frequency of events.
  </p>

    <ul>

      <li><strong>Use WebSockets for high-frequency bidirectionality:</strong> Chat apps (WhatsApp), co-authoring tools (Google Docs), and interactive gaming require WebSockets to avoid HTTP header bloat.</li>
      <li><strong>Decouple with Pub/Sub:</strong> Always explain how your stateful WebSocket servers communicate using an intermediate Redis Pub/Sub cluster to broadcast events across nodes.</li>
      <li><strong>Mitigate connection storms:</strong> Use exponential backoff with jitter to prevent your clients from crashing load balancers during service recovery.</li>
