<section className="content-section">
  <h2 id="core-problem">The Core Problem: Database Bloat</h2>
  <p>
    While relational databases support large data containers (such as PostgreSQL's `BYTEA` or MySQL's `LONGBLOB`), using them under high traffic creates critical system choke points.
  </p>
  <p>
    Relational database engines store records on disk inside fixed-size blocks (typically **8KB to 16KB pages**). When a column value exceeds the page size limit, the database is forced to store the value out-of-line in separate **overflow pages** (e.g. TOAST tables in PostgreSQL or off-page storage in InnoDB).
  </p>
  <p>
    This off-page serialization introduces massive performance penalties:
  </p>
  <ul>
    <li><strong>Buffer Pool Pollution:</strong> Database memory caches (the buffer pool) load pages to satisfy queries. Fetching rows with large embedded blobs drags massive byte arrays into memory, evicting key, frequently-read metadata indexes. This reduces the cache hit ratio of the entire database.</li>
    <li><strong>CPU & Network Exhaustion:</strong> Parsing, serializing, and pushing megabytes of base64/binary payloads through database connections spikes CPU cycles and saturates backend network interfaces.</li>
    <li><strong>Operational Fragility:</strong> Database backups, snapshot replication, and table maintenance (like `VACUUM`) slow down exponentially as database size scales into terabytes of binary blobs.</li>
  </ul>
</section>

<section className="content-section">
  <h2 id="storage-architectures">Storage Architectures Compared</h2>
  <p>
    When scaling large files, you must choose where the binary data actually lives:
  </p>

  <h3>1. In-Database BLOBs</h3>
  <p>
    The binary payload is saved in the database row alongside basic metadata. 
    <br/>
    <em>Verdict:</em> Simple to manage with transactional consistency, but causes severe write amplification and scales horribly. Use only for tiny assets (&lt; 100KB) that must change atomically with database state.
  </p>

  <h3>2. Local Application Filesystem</h3>
  <p>
    Files are written directly to local folders on the application server (e.g., `/var/www/uploads/`), and the filesystem path is saved in the database.
    <br/>
    <em>Verdict:</em> Highly performant read/write operations on SSDs, but breaks when scaling horizontally. Since application servers are stateless, subsequent requests routed to other servers will fail to find the file unless you use a network-attached filesystem (NFS/EFS), which introduces a single point of failure and severe locking lag.
  </p>

  <h3>3. Cloud Object Storage (S3 / GCS)</h3>
  <p>
    Files are stored as unique key-value objects in a flat namespace managed by a specialized cloud storage provider. The application database only saves the unique key/URI.
    <br/>
    <em>Verdict:</em> The industry-standard architecture. Durable (99.999999999% durability), highly available, cheap, and supports key protocols like CDN-caching, pre-signed authorization, and lifecycle rules.
  </p>
</section>

<section className="content-section">
  <h2 id="cdn-caching">CDN Caching & Edge Optimization</h2>
  <p>
    Once blobs are safely written to object storage, scaling reads is handled using a **Content Delivery Network (CDN)** like CloudFront or Cloudflare.
  </p>
  <p>
    When serving authenticated or premium media, you cannot expose static public S3 links. Instead, use these patterns:
  </p>
  <ul>
    <li><strong>Pre-signed URLs (CDN-Level):</strong> Similar to S3 pre-signed URLs, but signed using the CDN's private keys. The request hits edge nodes. If the signature is valid and the expiration is active, the CDN serves the cached file without hitting S3.</li>
    <li><strong>Pre-signed Cookies:</strong> Instead of signing individual file URLs (which breaks HLS/DASH video playlist files containing hundreds of individual segments), write a cookie with wildcard access parameters (e.g. `*.mp4` or `/movies/*`). The browser attaches this cookie to all subsequent media segment reads automatically.</li>
  </ul>
</section>

import software.amazon.awssdk.auth.credentials.DefaultCredentialsProvider;
import software.amazon.awssdk.regions.Region;
import software.amazon.awssdk.services.s3.S3Client;
import software.amazon.awssdk.services.s3.model.*;
import software.amazon.awssdk.services.s3.presigner.S3Presigner;
import software.amazon.awssdk.services.s3.presigner.model.*;

import java.time.Duration;
import java.util.ArrayList;
import java.util.List;

public class S3StorageManager {

    private final String bucketName = "production-app-blobs";
    private final Region region = Region.US_EAST_1;
    
    // 1. Generate standard pre-signed PUT Upload URL
    public String generateUploadPresignedUrl(String objectKey, String contentType) {
        try (S3Presigner presigner = S3Presigner.builder()
                .region(region)
                .credentialsProvider(DefaultCredentialsProvider.create())
                .build()) {
            
            PutObjectRequest putRequest = PutObjectRequest.builder()
                    .bucket(bucketName)
                    .key(objectKey)
                    .contentType(contentType)
                    .build();

            PutObjectPresignRequest presignRequest = PutObjectPresignRequest.builder()
                    .signatureDuration(Duration.ofMinutes(15))
                    .putObjectRequest(putRequest)
                    .build();

            PresignedPutObjectRequest presigned = presigner.presignPutObject(presignRequest);
            return presigned.url().toString(); // Direct S3 HTTP PUT endpoint
        }
    }

    // 2. Initiate Multipart Upload Session
    public MultipartSession initiateMultipart(String objectKey) {
        try (S3Client s3 = S3Client.builder().region(region).build()) {
            CreateMultipartUploadRequest request = CreateMultipartUploadRequest.builder()
                    .bucket(bucketName)
                    .key(objectKey)
                    .build();

            CreateMultipartUploadResponse response = s3.createMultipartUpload(request);
            return new MultipartSession(response.uploadId(), objectKey);
        }
    }

    // 3. Generate Pre-signed URL for a single part upload
    public String getPartPresignedUrl(String objectKey, String uploadId, int partNumber) {
        try (S3Presigner presigner = S3Presigner.builder().region(region).build()) {
            UploadPartRequest partRequest = UploadPartRequest.builder()
                    .bucket(bucketName)
                    .key(objectKey)
                    .uploadId(uploadId)
                    .partNumber(partNumber)
                    .build();

            UploadPartPresignRequest presignPartRequest = UploadPartPresignRequest.builder()
                    .signatureDuration(Duration.ofMinutes(20))
                    .uploadPartRequest(partRequest)
                    .build();

            PresignedUploadPartRequest presigned = presigner.presignUploadPart(presignPartRequest);
            return presigned.url().toString();
        }
    }

<section className="content-section">
  <h2 id="problem-breakdowns">Problem Breakdowns (Large Blobs in Practice)</h2>
  <p>
    Let's analyze how major applications design their ingestion, storage, and retrieval layers around large assets.
  </p>

  <h3>Dropbox / Google Drive (File Syncing)</h3>
  <ul>
    <li><strong>Problem:</strong> Syncing multi-gigabyte client files reliably while avoiding duplicating duplicate storage chunks.</li>
    <li><strong>Design Selection:</strong> **Content-Addressing Chunking & Global Deduplication**. Files are parsed client-side and split into fixed-size block chunks (e.g. 4MB). The client hashes each chunk (SHA-256). Before uploading, the client checks the hashes against the app metadata database. If a chunk already exists (uploaded by another user), the server skips the upload, saving bandwidth and storage. Uploads are coordinated chunk-by-chunk, enabling robust pause/resume functionality.</li>
  </ul>

  <h3>YouTube / Netflix (Video Ingestion & Playback)</h3>
  <ul>
    <li><strong>Problem:</strong> Ingesting raw 4K videos, encoding them for different bandwidth settings, and streaming them globally with zero buffering.</li>
    <li><strong>Design Selection:</strong> **Asynchronous Transcoding Pipelines + CDN segmenting**. Raw videos upload via multipart direct-to-S3 pre-signed requests. S3 triggers an asynchronous job (via EventBridge / AWS SQS) that starts a Fleet of worker nodes. These workers transcode the video into multiple resolutions (360p, 720p, 1080p, 4k) and chunk them into tiny 2-second segments (HLS/DASH formats). These segments are cached globally at CDN edge endpoints.</li>
  </ul>

  <h3>Instagram / Social Media (Image Feeds)</h3>
  <ul>
    <li><strong>Problem:</strong> Users uploading high-resolution camera images. Ingesting these files directly creates high latency and uses high mobile bandwidth.</li>
    <li><strong>Design Selection:</strong> **Client-Side Compression + Multi-Variant Asynchronous Ingestion**. The client application compresses and resizes the image to target dimensions (e.g. 1080px width) *before* triggering the upload. The client uploads the optimized image directly to Object Storage via a pre-signed URL. S3 triggers serverless functions (like AWS Lambda) to generate various resolution variants (thumbnails, compressed versions) and updates the feeds database.</li>
  </ul>
