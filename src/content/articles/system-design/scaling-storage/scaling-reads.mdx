<p>
  Most web applications have highly asymmetric traffic profiles. Social media platforms, news sites, and e-commerce catalogs frequently see read-to-write ratios of 95:5 or even 99:1. A system that serves 100,000 requests per second may only see 1,000 writes.
</p>
<p>
  If your primary database is responsible for processing both reads and writes, read queries will compete for connections and CPU cycles, eventually bottlenecking the database. This guide details how to scale your read capacity horizontally and maintain consistency in read-heavy architectures.
</p>

<section className="content-section">
  <h2 id="core-problem">The Core Problem</h2>
  <p>
    Databases typically write data to disk. Disk seeks are slow (taking milliseconds), whereas reading from memory takes nanoseconds. Even with indexes and memory buffers, a database's read capacity is restricted by:
  </p>
  <ul>
    <li><strong>Connection pool limits:</strong> Read queries block connections, forcing other requests to queue.</li>
    <li><strong>Disk I/O bottlenecks:</strong> Heavy join queries and aggregation filters pull data from cold disk storage, creating disk queues.</li>
    <li><strong>CPU saturation:</strong> Parsing SQL queries, sorting results, and evaluating complex index paths consumes database cores.</li>
  </ul>
</section>

<section className="content-section">

    To scale read capacity, we use a **Master-Replica (Primary-Secondary) database topology**. Writes are sent exclusively to the **Primary Database**, which processes them and propagates updates to one or more **Read Replicas**.
  </p>

  <p>
    Because replication is typically **asynchronous** to avoid slowing down writes, it introduces **Replication Lag**. If User A updates their username, and User B queries a replica 50ms later, User B will see the stale, pre-update username.
  </p>

  <h3 id="read-your-own-writes">Read-Your-Own-Writes Consistency</h3>
  <p>
    Replication lag is particularly jarring for the *writer*. If a user edits their comment, hits save, and the page refreshes, they expect to see their new comment. If the read hits a lagging replica, they see their old comment, leading them to think the update failed.
  </p>
  <p>
    To ensure **Read-Your-Own-Writes** consistency:
  </p>
  <ol>
    <li><strong>Session Pinning / Middleware:</strong> When a user performs a write, the server registers the write event (e.g., storing a timestamp in the user's session or cookie).</li>
    <li><strong>Conditional Routing:</strong> For any read queries within a short window (e.g., 5 seconds) after the write, the routing middleware forces the query to go to the **Primary Database** instead of a replica.</li>
    <li><strong>Replica Fallback:</strong> Once the window expires, reads fall back to replicas.</li>
  </ol>

<section className="content-section">
  <h2 id="caching-strategies">Multi-Level Caching</h2>
  <p>
    The best query is the one you never make to the database. Multi-level caching distributes read traffic across different layers of the infrastructure, closer to the user:
  </p>
  <ol>
    <li><strong>Client/Browser Cache:</strong> Storing static assets or session data in local memory or localStorage.</li>
    <li><strong>CDN (Content Delivery Network):</strong> Edge servers caching static media, documents, and public API responses globally.</li>
    <li><strong>API Gateway Cache:</strong> Caching endpoint outputs to prevent queries from hitting application servers.</li>
    <li><strong>In-Memory Cache (Redis/Memcached):</strong> Key-value memory stores serving database queries in &lt; 2ms.</li>
    <li><strong>In-Process Cache:</strong> Storing configuration files or static lookups directly in application server memory (heap) to avoid network calls entirely.</li>
  </ol>
</section>

<section className="content-section">
  <h2 id="precomputation-materialization">Pre-computation & Feed Generation</h2>
  <p>
    When reads require complex operations—such as joining multiple tables, calculating aggregations, or filtering feeds based on user relationships—running the query on demand is too slow.
  </p>
  <p>
    We scale this using **Pre-computation (Fan-out-on-write)**:
  </p>
  <ul>
    <li><strong>Fan-out-on-write:</strong> When a write occurs (e.g., a creator posts a video), a background worker pulls the creator's follower list and pre-injects the post ID into the feed caches of all active followers in Redis.</li>
    <li><strong>Read Path:</strong> When a follower reads their feed, they simply fetch their pre-computed list from Redis in $O(1)$ time, bypassing database joins.</li>
  </ul>
</section>

<section className="content-section">
  <h2 id="cqrs">CQRS Pattern</h2>
  <p>
    **Command Query Responsibility Segregation (CQRS)** separates the database models for write operations (Commands) from read operations (Queries).
  </p>
  <p>
    Under high read scale, relational tables designed for writes (normalized tables to prevent duplicate writes) are often highly inefficient for complex searches or filters.
  </p>

  <p>
    <strong>How it works:</strong>
  </p>
  <ol>
    <li>The write path uses a relational database optimized for consistency and normalized schemas.</li>
    <li>Every update triggers a Change Data Capture (CDC) event that streams modifications to a read-optimized search index (e.g. Elasticsearch) or a denormalized view table.</li>
    <li>The read path queries only the read-optimized store, enabling fast text searches and aggregations without impacting write performance.</li>
  </ol>
</section>

<section className="content-section">
  <h2 id="problem-breakdowns">Problem Breakdowns (Read Scaling in Practice)</h2>
  <p>
    Let's analyze how major system design problems apply read-scaling patterns.
  </p>

  <h3>Facebook News Feed (Feed Loading)</h3>
  <ul>
    <li><strong>Problem:</strong> Millions of concurrent users request their home feed. Running joins on posts and followers in SQL on every refresh is too slow.</li>
    <li><strong>Design Selection:</strong> **Pre-computed Caching (Fan-out-on-write) + CDN**. When a post is made, it is pushed to followers' feed cache arrays in Redis. Reads are simple key lookups. Public posts by celebrities with millions of followers use a hybrid "fan-out-on-read" approach to prevent write amplification.</li>
  </ul>

  <h3>Robinhood (Stock Price Streaming)</h3>
  <ul>
    <li><strong>Problem:</strong> Millions of users watching the same stock chart. The price database cannot handle millions of reads.</li>
    <li><strong>Design Selection:</strong> **SSE Edge Broadcast + In-Memory Hot Caching**. Stock prices are cached in an active Redis cluster. Instead of client polling, an API Gateway streams ticks to browsers using Server-Sent Events (SSE). The gateway reads from the hot cache, keeping database load near zero.</li>
  </ul>

  <h3>E-Commerce Catalog (Product Browsing)</h3>
  <ul>
    <li><strong>Problem:</strong> Serving millions of users browsing product descriptions that rarely change.</li>
    <li><strong>Design Selection:</strong> **CDN Caching + Cache-Aside (Redis)**. Because product descriptions change slowly, static HTML descriptions and image catalogs are cached at CDN edge nodes. Dynamic prices and stock levels are fetched via Cache-Aside from a Redis cluster.</li>
  </ul>
</section>
