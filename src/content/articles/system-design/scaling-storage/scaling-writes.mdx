<p>
  Unlike reads—which can be easily scaled using read replicas and content delivery networks (CDNs)—scaling write traffic is significantly more difficult. In relational databases (RDBMS), writes are bound to a single primary instance to guarantee ACID properties, making it a critical bottleneck.
</p>
<p>
  This guide covers how to scale write capacity by optimizing database storage engines (LSM vs B-Trees), buffering writes in application layers, using message queues to smooth out traffic spikes, and partitioning database nodes horizontally.
</p>

<section className="content-section">
  <h2 id="core-problem">The Core Problem</h2>
  <p>
    A standard database update requires modifying index pages on disk. In B-Tree indexes (the default for Postgres, MySQL, Oracle), these modifications translate to **random disk I/O seeks**, which are extremely expensive.
  </p>
  <p>
    Under high write throughput, databases crash due to:
  </p>
  <ul>
    <li><strong>Disk write exhaustion:</strong> Disk queues fill up as write heads jump across sector ranges.</li>
    <li><strong>Lock Contention:</strong> Transactions wait to acquire write locks on shared rows and index nodes.</li>
    <li><strong>Transaction log saturation:</strong> The Write-Ahead Log (WAL) disk partition is choked by transaction serialization tasks.</li>
  </ul>
</section>

  <p>
    <strong>How LSM Trees work:</strong>
  </p>
  <ol>
    <li><strong>Commit Log (WAL):</strong> Incoming writes are logged sequentially to an append-only file on disk. This is extremely fast because disk heads do not jump.</li>
    <li><strong>MemTable:</strong> The write is immediately stored in memory in a sorted structure (e.g. a skip list). The write operation is acknowledged as complete.</li>
    <li><strong>SSTables (Sorted String Tables):</strong> Once the MemTable fills up, it is flushed to disk sequentially as an SSTable file.</li>
    <li><strong>Compaction:</strong> Background threads periodically merge smaller SSTables into larger ones, cleaning up duplicate edits and deleted entries.</li>
  </ol>
  
  <Callout type="tip">
    <p>
      For systems requiring high-throughput event logging, telemetry, or metric monitoring, always select an LSM-based database like Cassandra or InfluxDB. Do not use relational databases with B-Tree indexes unless write volume is low.
    </p>
  </Callout>

from sqlalchemy.orm import Session
from sqlalchemy import insert
import threading
import time

class WriteBuffer:
    def __init__(self, session_maker, batch_size=1000, max_interval=5.0):
        self.session_maker = session_maker
        self.batch_size = batch_size
        self.max_interval = max_interval
        self.buffer = []
        self.lock = threading.Lock()
        self.last_flush = time.time()

    def add_record(self, metric_data: dict):
        with self.lock:
            self.buffer.append(metric_data)
            if len(self.buffer) >= self.batch_size:
                self._flush()

    def check_timer(self):
        with self.lock:
            if self.buffer and (time.time() - self.last_flush) >= self.max_interval:
                self._flush()

    def _flush(self):
        if not self.buffer:
            return
        session = self.session_maker()
        try:
            # Executes a single bulk insert statement
            session.execute(insert(Metric), self.buffer)
            session.commit()

            raise e

<section className="content-section">
  <h2 id="sharding-scaling">Horizontal Sharding</h2>
  <p>
    If a single database machine runs out of resources, we must split the data across multiple machines. This is **Horizontal Database Sharding**.
  </p>
  <p>
    Every write request is routed to a specific physical server based on a shard key:
    `target_db = hash(shard_key) % N`
  </p>
  <p>
    For example, if you shard an order database by `user_id`, all orders for User A go to Shard 1, while User B's orders go to Shard 2.
  </p>
  
  <Callout type="info">
    <p>
      Database sharding is a massive architectural shift. For a deep dive into sharding strategies, shard key selection, re-sharding, and maintaining cross-shard constraints, see our dedicated <strong>Sharding</strong> guide.
    </p>
  </Callout>
</section>

<section className="content-section">
  <h2 id="problem-breakdowns">Problem Breakdowns (Write Scaling in Practice)</h2>
  <p>
    Let's analyze how major system design problems apply write-scaling patterns.
  </p>

  <h3>Metrics Monitoring (Datadog / Prometheus)</h3>
  <ul>
    <li><strong>Problem:</strong> Millions of servers pushing CPU and memory metrics every second. Writes are massive, append-only, and time-sequential.</li>
    <li><strong>Design Selection:</strong> **LSM Time-Series Databases (InfluxDB) + Kafka**. We buffer incoming metrics at the edge using Kafka. Background worker clusters pull metric records, group them by server/metric, and perform bulk writes to an LSM-based time-series database. B-Tree relational databases would collapse immediately under the random index writes.</li>
  </ul>

  <h3>Uber (Driver GPS Tracking)</h3>
  <ul>
    <li><strong>Problem:</strong> Hundreds of thousands of active drivers sending location updates every 4 seconds.</li>
    <li><strong>Design Selection:</strong> **Write-Back Buffer (Redis) + NoSQL Cassandra**. GPS coordinates are written to a fast, in-memory Redis cluster to update driver location immediately (reads for passenger maps). A background job streams updates from Redis to Cassandra (LSM structure) using batching, ensuring driver location histories are persisted sequentially.</li>
  </ul>

  <h3>E-Commerce Flash Sale (Order Volume)</h3>
  <ul>
    <li><strong>Problem:</strong> A product release triggers 100,000 orders in the first 2 seconds, overloading the order relational database.</li>
    <li><strong>Design Selection:</strong> **Queue-Based Load Leveling (Kafka)**. The API gateway does not write directly to the database. It validates basic user formats and drops order payloads onto a Kafka queue partition. Workers pull orders from Kafka at a controlled rate (e.g. 2,000 orders/sec) to execute relational writes safely without exhausting connection pools.</li>
  </ul>
</section>
