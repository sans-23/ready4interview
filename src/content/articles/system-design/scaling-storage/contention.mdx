<section className="content-section">
  <h2 id="core-problem">The Core Problem</h2>
  <p>
    When two transactions read the same data, calculate updates in memory, and write them back, one write inevitably overwrites the other. This is the **Lost Update Problem**.
  </p>
  <p>
    If the database resolves this by locking the row, subsequent transactions are blocked. Under high write traffic, this blocking causes:
  </p>
  <ul>
    <li><strong>Connection Pool Exhaustion:</strong> Threads stack up waiting for the row lock, consuming server connections until the pool is empty and all other operations fail.</li>
    <li><strong>High Latency:</strong> The tail latency (p99) rises exponentially as clients wait in line.</li>
    <li><strong>Deadlocks:</strong> Concurrent transactions locking multiple resources in different orders crash the database engine.</li>
  </ul>
</section>

public interface InventoryRepository extends JpaRepository<Inventory, Long> {

from sqlalchemy.orm import Session
from sqlalchemy.orm.exc import StaleDataError

class Product(Base):
    __tablename__ = 'products'
    id = Column(Integer, primary_key=True)
    stock = Column(Integer)
    # SQLAlchemy natively supports OCC versioning via version_id_col
    version = Column(Integer, nullable=False)
    __mapper_args__ = {
        "version_id_col": version
    }

def purchase_product(session: Session, product_id: int, quantity: int):
    for attempt in range(3):
        try:
            product = session.query(Product).filter_by(id=product_id).first()
            if product.stock < quantity:
                raise ValueError("Out of stock")
                
            product.stock -= quantity
            session.commit()  # Throws StaleDataError if database row version differs

  <h3 id="atomic-updates">Database Atomic Updates</h3>
  <p>
    Many systems do not need to read, check, and write in separate operations. Instead, they can execute atomic mutations directly on the database engine.
  </p>
  <p>
    `UPDATE products SET stock = stock - 1 WHERE id = 123 AND stock >= 1;`
  </p>
  <p>
    This delegates concurrency safety to the database engine. The database will acquire an exclusive row-level lock on the targeted row during the update, perform the math in memory, and immediately release it.
  </p>
  <p>
    <strong>Tradeoffs:</strong>
  </p>
  <ul>
    <li><strong>Pros:</strong> Eliminates application-side OCC retry loops. Extremely fast and supported by virtually all databases (Postgres, MySQL, DynamoDB, MongoDB).</li>
    <li><strong>Cons:</strong> Limited to simple mathematical operations. You cannot execute complex business logic (e.g., checking user tiers or validating discounts) within a single update statement.</li>
  </ul>

          <div style={{ fontSize: '0.72rem', color: 'var(--text-muted)', marginTop: '4px' }}>
            User ID Hash % N routes to a specific slot
          </div>
        </div>

        <div style={{ background: '#e3f2fd', border: '1px solid #90caf9', padding: '12px', borderRadius: '8px', textAlign: 'center', width: '100%', maxWidth: '220px' }}>
          <strong>Read Query (Aggregator)</strong>
          <div style={{ fontSize: '0.8rem', color: '#1565c0', fontWeight: 'bold', marginTop: '6px' }}>
            SELECT SUM(count) FROM slots WHERE entity_id = X
          </div>
        </div>
      </div>

    </div>
  </div>

  <p>
    <strong>How it works:</strong>
  </p>
  <ul>
    <li><strong>Write Path:</strong> When a user likes a post, the app hashes the user's ID modulo $N$ (e.g., $N=10$). It then increments only that specific slot:
    `UPDATE post_likes SET count = count + 1 WHERE post_id = 123 AND slot = 4;`</li>
    <li><strong>Read Path:</strong> To display the total count, query all slots and sum them:
    `SELECT SUM(count) FROM post_likes WHERE post_id = 123;`</li>
  </ul>
  <p>
    This spreads the write pressure across 10 distinct records instead of 1, effectively scaling write throughput by a factor of $10$.
  </p>
</section>

<section className="content-section">
  <h2 id="problem-breakdowns">Problem Breakdowns (Contention in Practice)</h2>
  <p>
    Let's analyze how major system design problems handle high write contention.
  </p>

  <h3>Ticketmaster / Flight Booking (Seat Allocation)</h3>
  <ul>
    <li><strong>Problem:</strong> Thousands of users locking the exact same seat concurrently in the final seconds of ticket releases.</li>
    <li><strong>Design Selection:</strong> **Pessimistic Locking with Reservation Timeouts**. Use database row-level locking (`SELECT ... FOR UPDATE`) to temporarily reserve the seat for 10 minutes. If payment completes, the lock is committed. If the timeout expires, a cron job releases the reservation. Pessimistic locking is chosen here because consistency is critical: selling the same seat twice is a severe business failure.</li>
  </ul>

  <h3>Flash Sale / E-Commerce (Stock Ingestion)</h3>
  <ul>
    <li><strong>Problem:</strong> 100 available iPhones are targeted by 100,000 requests per second.</li>
    <li><strong>Design Selection:</strong> **Redis Reservation + Asynchronous Queueing**. Do not hit the relational database. Load the stock into an in-memory Redis cluster (`stock = 100`). Use a **Redis Lua Script** (single-threaded execution) to check and decrement stock:
    `redis.call("DECRBY", KEYS[1], ARGV[1])`. If the result is &ge; 0, place the order on a message queue and return `202 Accepted`. A single-threaded worker reads the queue and processes orders sequentially.</li>
  </ul>

  <h3>Online Auction (Bidding Engine)</h3>
  <ul>
    <li><strong>Problem:</strong> Bidders competing for a rare item. Bids must be strictly higher than the current bid, requiring sub-second updates and strong serialization.</li>
    <li><strong>Design Selection:</strong> **Distributed Lock (Redis Redlock / ZooKeeper Fencing Tokens)**. Protect the auction balance using a distributed lock tied to the auction ID. Before processing a bid, a thread acquires the lock, checks if the bid exceeds the current leader in Redis/DB, writes the new high bid, and releases the lock.</li>
  </ul>

  <h3>Social Post Likes (Viral Counters)</h3>
  <ul>
    <li><strong>Problem:</strong> A post by a celebrity gets millions of likes in minutes.</li>
    <li><strong>Design Selection:</strong> **In-Memory Buffering (Write-back Caching) + Partitioned Counters**. The write path does not modify the database immediately. Instead, local application servers buffer count events in memory or aggregate them in Redis. Every 5 seconds, the accumulated writes are flushed to a database configured with **Partitioned Counters** to distribute the SQL update pressure.</li>
  </ul>

  <h3>Distributed Wallet / Ledger (Transfers)</h3>
  <ul>
    <li><strong>Problem:</strong> High-frequency financial transfers modifying a single user's wallet balance.</li>
    <li><strong>Design Selection:</strong> **Double-Entry Bookkeeping (Append-Only Ledger)**. Avoid running `UPDATE balance = balance - X`. Instead, treat writes as append-only log transactions (`INSERT INTO transactions ...`). Calculating the balance is a read aggregation. If an active balance constraint is needed, combine with Optimistic Concurrency Control (OCC) or memory reservations in Redis.</li>
  </ul>
</section>

  <h2 id="conclusion">Conclusion</h2>
  <p>
    Handling contention is about moving along the spectrum of strict consistency and performance. In interviews, begin by defining the rate of write operations on a single resource, then justify your locking or serialization selection.
  </p>

      Key Interview Takeaways
    </h3>
    <ul>
      <li><strong>Pessimistic locking is for high-stakes consistency:</strong> If duplicate updates cause critical financial or booking failures, use `SELECT ... FOR UPDATE` despite the latency cost.</li>
      <li><strong>Atomic DB updates are the fastest win:</strong> When you only need relative math on a row, use `UPDATE SET x = x + 1` directly to let the database handle the row lock.</li>
      <li><strong>Use distributed locks with lease timeouts:</strong> Always mention fencing tokens or short TTL leases when using Redis distributed locks to prevent deadlocks on node crashes.</li>
      <li><strong>Scale counters with slots:</strong> If a single counter is the bottleneck, scale write throughput by distributing updates across $N$ sub-counters (Partitioned Counters).</li>
    </ul>
