    navigator.clipboard.writeText(activeCode);

            background: 'none',
            border: '1px solid rgba(255, 255, 255, 0.1)',

<section className="content-section">
  <h2 id="class-loader-subsystem">Class Loader: Delegation & Linking</h2>
  <p>
    The **ClassLoader Subsystem** dynamically loads class files into the JVM at runtime. It operates in three phases: **Loading**, **Linking**, and **Initialization**.
  </p>

  <h3>1. Loading Phase</h3>
  <p>
    Fetches the binary stream of the `.class` file by name and constructs its representation in the Method Area, generating a `java.lang.Class` object on the Heap.
  </p>

  <h3>2. Linking Phase</h3>
  <ul>
    <li>**Verify:** Validates the compiled bytecode against the JVM specifications to ensure it does not violate security restrictions or contain corrupted instructions.</li>
    <li>**Prepare:** Allocates memory for static fields and initializes them to default type values (e.g. references to `null`, integers to `0`). Values defined in code are not assigned yet.</li>
    <li>**Resolve:** Replaces symbolic references in the runtime constant pool with direct memory addresses.</li>
  </ul>

  <h3>3. Initialization Phase</h3>
  <p>
    Executes the static initializers (running `<clinit>` methods) and assigns the values defined in source code to static variables.
  </p>

  <h3>ClassLoader Delegation Hierarchy</h3>
  <p>
    Java resolves class paths using the **Delegation Principle**:
  </p>
  <div style={{ background: '#1a1b26', padding: '16px', borderRadius: '8px', fontFamily: 'monospace', fontSize: '0.85rem', color: '#a9b1d6', border: '1px solid var(--border-color)', margin: '20px 0', lineHeight: '1.6' }}>
    Bootstrap ClassLoader (Core Java APIs - rt.jar / java.base)<br/>
    &nbsp;&nbsp;&nbsp;&nbsp;▲<br/>
    Platform ClassLoader (Extension APIs - platform modules)<br/>
    &nbsp;&nbsp;&nbsp;&nbsp;▲<br/>
    Application ClassLoader (ClassPath & App jars)<br/>
  </div>
  <p>
    When a class loader receives a loading request, it delegates it to its parent loader first. The class is loaded locally only if the parent loader cannot find it. This prevents user-defined classes from overriding core platform classes (e.g., preventing a malicious `java.lang.String` from being loaded).
  </p>
</section>

<section className="content-section">
  <h2 id="runtime-data-areas">Memory Areas: Heap, Stack & MetaSpace</h2>
  <p>
    The JVM partitions its memory into thread-shared and thread-private data areas.
  </p>

  <h3>1. Thread-Shared Memory (Global Access)</h3>
  <ul>
    <li>
      <strong>Heap Area:</strong> The main memory area where all object instances, arrays, and their instance variables are allocated. The Heap is the primary target of Garbage Collection.
    </li>
    <li>
      <strong>Method Area (MetaSpace):</strong> Stores class structures, metadata, method details, constant pools, and static variables. In Java 8+, this was refactored from the fixed-size heap **PermGen** to **MetaSpace**, which dynamically allocates memory from native OS RAM, reducing `OutOfMemoryError` limits.
    </li>
  </ul>

  <h3>2. Thread-Private Memory (Created per Thread)</h3>
  <ul>
    <li>
      <strong>JVM Stack:</strong> Stores stack frames. A new frame is pushed onto the stack for every method call and popped off when the method returns. A frame contains:
      <ul>
        <li>**Local Variable Table (LVT):** Stores method parameters and local variables.</li>
        <li>**Operand Stack:** Holds intermediate values during mathematical operations.</li>
        <li>**Frame Data:** Holds references to the runtime constant pool and exception tables.</li>
      </ul>
    </li>
    <li>
      <strong>PC (Program Counter) Register:</strong> Holds the address of the current JVM instruction being executed by the thread.
    </li>
    <li>
      <strong>Native Method Stack:</strong> Holds instruction contexts for JNI native calls.
    </li>
  </ul>
</section>

<section className="content-section">
  <h2 id="execution-engine">Execution Engine: JIT & Tiered Compilers</h2>
  <p>
    The Execution Engine executes compiled bytecode by translating it into native machine instructions.
  </p>

  <h3>1. Interpreter vs. JIT Compiler</h3>
  <ul>
    <li>**Interpreter:** Reads and executes bytecode instructions one by one. It starts quickly but runs slowly due to repeated parsing.</li>
    <li>**Just-In-Time (JIT) Compiler:** Identifies "hot spots" (frequently executed methods and loops) and compiles their bytecode directly into native machine code for faster execution.</li>
  </ul>

  <h3>2. Tiered Compilation (Default Engine)</h3>
  <p>
    Modern JVMs use **Tiered Compilation** (combining the speed of the C1 compiler with the deep optimization of the C2 compiler):
  </p>
  <ul>
    <li>**Tier 0 (Interpreter):** Bytecode execution begins here. It profiles method execution to identify hot spots.</li>
    <li>**Tier 1-3 (C1 Client Compiler):** Compiles bytecode to native code quickly, applying basic optimizations like method inlining and loop unrolling.</li>
    <li>**Tier 4 (C2 Server Compiler):** Handles highly optimized native compilation. It applies advanced optimizations such as global escape analysis, lock elision (removing unnecessary synchronization), and dead code removal.</li>
  </ul>
</section>

    Understanding JVM internals is essential for optimizing application performance. Analyzing ClassLoader delegation hierarchies, managing runtime memory splits (Heap vs Stack vs MetaSpace), and utilizing JIT tiered compilers are core concepts in JVM tuning.

      Key Interview Takeaways
    </h3>
    <ul>
      <li><strong>Explain the ClassLoader Delegation Principle:</strong> Requests are delegated to parent class loaders first. This ensures core JVM APIs (like `java.lang.String`) cannot be overridden by user jars.</li>
      <li><strong>What are the ClassLoader linking steps?</strong> Verification (verifies byte syntax security), Preparation (allocates static field default values), and Resolution (resolves symbolic constant references).</li>
      <li><strong>What is the difference between Heap and Stack?</strong> The Heap is shared across threads and holds object allocations. Stacks are private to each thread and hold short-lived method parameters and local variable frames.</li>
      <li><strong>How does MetaSpace differ from PermGen?</strong> PermGen was a fixed-size heap space. MetaSpace resides in off-heap native memory, growing dynamically to prevent fixed-size `OutOfMemoryError` conditions.</li>
      <li><strong>What is Tiered Compilation?</strong> The JVM starts by running interpreted code (Tier 0). As hotspots are identified, they are compiled to native code using the C1 client compiler (Tiers 1-3) and then further optimized using the C2 server compiler (Tier 4) for peak performance.</li>
    </ul>
