<section className="content-section">
  <h2 id="cas-mechanics">CAS: Optimistic Lock-Free Concurrency</h2>
  <p>
    **Compare-And-Swap (CAS)** is an optimistic concurrency control mechanism that operates at the hardware level. Rather than blocking other threads to update a variable, a thread attempts the update lock-free, spinning and retrying if another thread modifies the variable first.
  </p>
  <p>
    CAS operations accept three arguments:
  </p>
  <ul>
    <li>Memory Location (`V`): The physical memory address of the variable.</li>
    <li>Expected Old Value (`A`): The value the thread believes is currently at address `V`.</li>
    <li>New Value (`B`): The value the thread wishes to write to address `V`.</li>
  </ul>
  <p>
    The CPU updates `V` to `B` if and only if the current value at `V` equals `A` (atomic execution). Modern processors compile this directly to single assembly-level instructions (e.g. `cmpxchg` on x86 architectures), which bypasses kernel locks.
  </p>

  <h3>The ABA Problem</h3>
  <p>
    A classic challenge with CAS is the **ABA problem**:
  </p>
  <ol>
    <li>Thread 1 reads variable value `A` from memory.</li>
    <li>Thread 2 preempts Thread 1, updates the value from `A` to `B`, and then changes it back to `A`.</li>
    <li>Thread 1 resumes, executes CAS expecting `A`, sees `A`, and succeeds.</li>
  </ol>
  <p>
    Although the value appears unchanged to Thread 1, the state was modified. This can lead to issues in pointer-linked structures, such as nodes being reallocated.
  </p>
  <p>
    <strong>Solution:</strong> Use version stamping. Java provides `AtomicStampedReference`, which pairs the object reference with an integer stamp (version number). The CAS operation succeeds only if both the reference and the stamp match the expected values.
  </p>
</section>

<section className="content-section">
  <h2 id="aqs-internals">AQS: The Engine Behind Java Locks</h2>
  <p>
    The **AbstractQueuedSynchronizer (AQS)** class is the foundation of Java's lock framework. It is used to implement synchronizers like `ReentrantLock`, `Semaphore`, `CountDownLatch`, and `ReentrantReadWriteLock`.
  </p>

  <h3>AQS Core Components</h3>
  <ul>
    <li>
      <strong>Volatile State:</strong> A single 32-bit integer, `private volatile int state`, representing lock ownership status.
      <ul>
        <li>For `ReentrantLock`: `0` indicates unlocked, while `1` or greater represents locked (supporting nested reentrant locks).</li>
        <li>For `Semaphore`: Represents the number of available permits.</li>
        <li>For `CountDownLatch`: Represents the current countdown gate value.</li>
      </ul>
    </li>
    <li>
      <strong>CLH FIFO Wait Queue:</strong> A doubly-linked list of `Node` elements holding blocked threads.
    </li>
  </ul>

  {/* AQS Layout Visual */}
  <div style={{ background: '#1a1b26', padding: '20px', borderRadius: '8px', fontFamily: 'monospace', fontSize: '0.85rem', color: '#a9b1d6', border: '1px solid var(--border-color)', margin: '24px 0', lineHeight: '1.6' }}>
    <span style={{ color: '#7c3aed', fontWeight: 'bold' }}>AQS Structure:</span><br/>
    ├── [volatile int state] (Updates via CAS: compareAndSetState)<br/>
    └── [CLH Queue] (Doubly-Linked List)<br/>
        &nbsp;&nbsp;&nbsp;&nbsp;[Head Node] &lt;──► [Blocked Thread Node] &lt;──► [Tail Node]<br/>
  </div>

  <h3>Lock Acquisition Mechanics</h3>
  <ol>
    <li>A thread attempts to acquire the lock by executing CAS to change `state` (e.g. from 0 to 1).</li>
    <li>If CAS succeeds, the thread sets itself as the lock owner and proceeds.</li>
    <li>If CAS fails (indicating the lock is held), the thread is wrapped in a `Node` and appended to the tail of the CLH queue.</li>
    <li>Once queued, the thread is suspended using `LockSupport.park()`, which puts it into the `WAITING` or `BLOCKED` state.</li>
    <li>When the lock owner releases it, it resets the `state` via volatile write and calls `LockSupport.unpark()` on the successor thread at the head of the queue.</li>
  </ol>
</section>

<section className="content-section">
  <h2 id="reentrantlock-features">ReentrantLock: Beyond synchronized</h2>
  <p>
    `ReentrantLock` implements the `Lock` interface, providing explicit lock control as an alternative to implicit `synchronized` blocks.
  </p>
  
  <p>
    <strong>Key features of ReentrantLock:</strong>
  </p>
  <ul>
    <li>
      <strong>Fairness Policies:</strong> A `synchronized` block does not guarantee acquisition order, which can lead to thread starvation. `ReentrantLock` supports an optional fairness parameter:
      <br/>
      `new ReentrantLock(true)`
      <br/>
      Fair locks grant access to the longest-waiting thread. Note that this reduces throughput due to the extra tracking overhead.
    </li>
    <li>
      <strong>Non-blocking lock attempts:</strong> Using `tryLock()`, a thread can attempt lock acquisition. If the lock is held, it returns `false` immediately instead of blocking, allowing the thread to perform other tasks.
    </li>
    <li>
      <strong>Interruptible Locks:</strong> A thread waiting for a lock via `lockInterruptibly()` can be interrupted from its waiting state, throwing an `InterruptedException`.
    </li>
    <li>
      <strong>Multiple Condition Queues:</strong> `synchronized` provides only a single wait queue per object (via `wait()` and `notify()`). `ReentrantLock` allows creating multiple `Condition` objects:
      <br/>
      `Condition notFull = lock.newCondition();`
      <br/>
      This enables separating producer and consumer wait queues, allowing `signal()` to target specific thread groups.
    </li>
  </ul>
</section>

<section className="content-section">
  <h2 id="stampedlock-optimistic">StampedLock: Optimistic Read Scaling</h2>
  <p>
    While `ReentrantReadWriteLock` improves performance by allowing multiple concurrent readers, it has a drawback: if readers are constant, writers can be starved indefinitely.
  </p>
  <p>
    To address this, Java 8 introduced `StampedLock`. It supports three lock modes: Write, Read, and **Optimistic Read**.
  </p>

  <h3>Optimistic Reading Mechanics</h3>
  <p>
    Unlike standard read locks, an optimistic read does not acquire a shared lock. It does not write to a volatile state or block other threads. Instead, it reads a version stamp:
  </p>

```java
long stamp = lock.tryOptimisticRead(); // Returns stamp, does not block writers
// Read fields into local variables
if (!lock.validate(stamp)) { // Check if a write occurred since reading the stamp
    stamp = lock.readLock(); // Fallback: acquire a blocking read lock
    try {
        // Read fields again safely
    } finally {
        lock.unlockRead(stamp);
    }
}
```

  <p>
    If `validate(stamp)` returns true, no write occurred during the read, and the local variables are safe to use. If a write did occur, the validation fails, and the thread falls back to acquiring a blocking read lock.
  </p>
  <p>
    This pattern eliminates read lock contention in read-heavy, write-light workloads.
  </p>
</section>

import java.util.Queue;
import java.util.locks.Condition;
import java.util.locks.Lock;
import java.util.locks.ReentrantLock;

public class BoundedLockBuffer<T> {
    private final Queue<T> queue = new LinkedList<>();
    private final int capacity;
    
    private final Lock lock = new ReentrantLock();
    // Dual condition variables bound to the same lock instance
    private final Condition notFull = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();

    public BoundedLockBuffer(int capacity) {
        this.capacity = capacity;
    }

    public void put(T item) throws InterruptedException {
        lock.lock();
        try {
            while (queue.size() == capacity) {
                notFull.await(); // Producer waits on notFull condition
            }
            queue.add(item);
            // Signal only consumer threads waiting on notEmpty
            notEmpty.signal(); 
        } finally {
            lock.unlock(); // Ensure lock is released in finally block
        }
    }

    public T take() throws InterruptedException {
        lock.lock();
        try {
            while (queue.isEmpty()) {
                notEmpty.await(); // Consumer waits on notEmpty condition
            }
            T item = queue.poll();
            // Signal producer threads waiting on notFull
            notFull.signal(); 
            return item;
        } finally {
            lock.unlock();
        }
    }
}`} />

public class Point2D {
    private double x, y;
    private final StampedLock lock = new StampedLock();

    public void move(double deltaX, double deltaY) {
        long stamp = lock.writeLock(); // Acquire exclusive write lock
        try {
            x += deltaX;
            y += deltaY;
        } finally {
            lock.unlockWrite(stamp);
        }
    }

    // High performance read using StampedLock Optimistic Validation
    public double distanceFromOrigin() {
        long stamp = lock.tryOptimisticRead(); // Non-blocking stamp read
        double currentX = x;
        double currentY = y;
        
        // Validate if a write occurred during the read
        if (!lock.validate(stamp)) {
            stamp = lock.readLock(); // Fallback to standard blocking read lock
            try {
                currentX = x;
                currentY = y;
            } finally {
                lock.unlockRead(stamp);
            }
        }
        return Math.sqrt(currentX * currentX + currentY * currentY);

      Key Interview Takeaways

    <ul>
      <li><strong>What is AQS?</strong> The AbstractQueuedSynchronizer is a framework that uses a volatile state integer and a FIFO queue of blocked threads to implement locks and synchronizers.</li>
      <li><strong>Explain the ABA problem and its solution:</strong> The ABA problem occurs when a value changes from A to B and back to A, making it appear unchanged to a CAS operation. The solution is version stamping (e.g., using `AtomicStampedReference`).</li>
      <li><strong>How do ReentrantLock Conditions work?</strong> They allow creating multiple distinct wait sets on a single lock, allowing threads to target specific notification queues (e.g., separating `notEmpty` and `notFull` signals).</li>
      <li><strong>What is StampedLock optimistic reading?</strong> A non-blocking read mode that reads a version stamp without acquiring a lock. It validates the stamp afterwards, falling back to a standard read lock only if a write occurred during the read.</li>
      <li><strong>Is StampedLock reentrant?</strong> No, StampedLock is not reentrant. Attempting to acquire a lock you already hold will result in a deadlock.</li>
    </ul>
