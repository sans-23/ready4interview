    navigator.clipboard.writeText(activeCode);

            background: 'none',
            border: '1px solid rgba(255, 255, 255, 0.1)',

<section className="content-section">
  <h2 id="chm-internals">ConcurrentHashMap: Bucket Locks & Resizing</h2>
  <p>
    `ConcurrentHashMap` is Java's high-performance thread-safe map. Its design has evolved to support high read/write concurrency.
  </p>

  <h3>1. Evolution of Locking Mechanisms</h3>
  <ul>
    <li>
      <strong>Java 7: Segment Locking:</strong> The map was divided into an array of 16 segments, each acting as an independent hashtable with its own lock. A write only locked one segment, allowing threads to write to other segments concurrently. However, writes to the same segment were blocked.
    </li>
    <li>
      <strong>Java 8+: Bucket-level Locking:</strong> Segment arrays were replaced with a flat array of volatile nodes, identical to `HashMap`. Write operations (`put`) synchronize only on the **head node** of the target bucket. Threads can write to different buckets concurrently, improving write scalability.
    </li>
  </ul>

  {/* ConcurrentHashMap layout */}
  <div style={{ background: '#1a1b26', padding: '20px', borderRadius: '8px', fontFamily: 'monospace', fontSize: '0.85rem', color: '#a9b1d6', border: '1px solid var(--border-color)', margin: '24px 0', lineHeight: '1.6' }}>
    <span style={{ color: '#7c3aed', fontWeight: 'bold' }}>CHM Node Array:</span> [Bucket 0] [Bucket 1] [Bucket 2] ... [Bucket N]<br/>
    &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;│<br/>
    &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;▼ (Synchronized on head node ONLY for writes)<br/>
    &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;[Node A] ──► [Node B] (Linked list or Red-Black Tree)<br/>
  </div>

  <h3>2. Lock-Free Volatile Reads</h3>
  <p>
    Read operations (`get`) are **completely lock-free**. The nodes in the backing array are declared `volatile`, ensuring that any write to a node is immediately visible to reader threads. Reads traverse bucket structures without acquiring locks, eliminating read contention.
  </p>

  <h3>3. Cooperative Multi-Threaded Resizing</h3>
  <p>
    When a resize is triggered, a single thread begins the transition. If other threads attempt writes while a resize is in progress:
  </p>
  <ul>
    <li>They encounter a special `ForwardingNode` in the bucket they are targeting, which redirects lookup requests to the new table.</li>
    <li>Instead of blocking, these threads cooperate in the resizing process by calling `helpTransfer()`, assisting in copying other buckets to the new table.</li>
  </ul>
</section>

<section className="content-section">
  <h2 id="copyonwrite-structures">CopyOnWrite: Snapshot Read Scaling</h2>
  <p>
    `CopyOnWriteArrayList` maintains a volatile reference to an array: `private transient volatile Object[] array;`.
  </p>

  <h3>1. Write-Copy Mechanics</h3>
  <p>
    Any mutative operation (`add`, `set`, `remove`) uses an exclusive lock to perform the following steps:
  </p>
  <ol>
    <li>Copy the existing array to a new, larger array.</li>
    <li>Apply the modification to the new array copy.</li>
    <li>Point the volatile `array` reference to the new array.</li>
    <li>Release the lock.</li>
  </ol>
  <pre style={{ background: '#1a1b26', padding: '16px', borderRadius: '8px', overflowX: 'auto', color: '#e2e8f0', border: '1px solid var(--border-color)', fontSize: '0.85rem' }}>
{`// Simplified representation of CopyOnWriteArrayList write
public boolean add(E e) {
    synchronized (lock) {
        Object[] es = getArray();
        int len = es.length;
        Object[] newElements = Arrays.copyOf(es, len + 1);
        newElements[len] = e;
        setArray(newElements);
        return true;
    }
}`}
  </pre>

  <h3>2. Lock-Free Snapshot Reads</h3>
  <p>
    Read operations do not acquire locks. They read from the current array reference. Because the array is never modified in place, readers can iterate over the snapshot safely without encountering a `ConcurrentModificationException` or needing lock protection.
  </p>
  <p>
    <strong>Suitability:</strong> This structure is ideal for read-heavy, write-rare scenarios (e.g. system configurations, listener arrays). It is inefficient for frequent writes due to the overhead of copying the entire array.
  </p>
</section>

<section className="content-section">
  <h2 id="blocking-queues">BlockingQueues: Coordinator Hand-offs</h2>
  <p>
    `BlockingQueue` implementations are designed for producer-consumer pipelines, blocking threads when the queue is full or empty.
  </p>

  <h3>1. ArrayBlockingQueue vs. LinkedBlockingQueue</h3>
  <ul>
    <li>
      <strong>`ArrayBlockingQueue` (Bounded):</strong> Backed by a fixed-size array. It uses a **single lock** (`ReentrantLock`) to coordinate both reads and writes. This means producers and consumers block each other, which can limit throughput in high-contention systems.
    </li>
    <li>
      <strong>`LinkedBlockingQueue` (Optionally Bounded):</strong> Backed by linked nodes. It uses a **two-lock** algorithm, maintaining separate locks for takes (`takeLock`) and puts (`putLock`). This allows a producer and a consumer to operate concurrently, increasing throughput.
    </li>
  </ul>

  <h3>2. SynchronousQueue & ConcurrentLinkedQueue</h3>
  <ul>
    <li>
      <strong>`SynchronousQueue` (Zero Capacity):</strong> A handoff queue where each insert must wait for a corresponding take. It holds no elements, acting as a direct transfer mechanism between threads.
    </li>
    <li>
      <strong>`ConcurrentLinkedQueue` (Lock-Free):</strong> A non-blocking FIFO queue backed by linked nodes. It uses CAS instructions instead of locks, making it ideal for high-throughput, non-blocking producer-consumer scenarios.
    </li>
  </ul>
</section>

import java.util.concurrent.LinkedBlockingQueue;
import java.util.concurrent.TimeUnit;

public class ProducerConsumerPipeline {
    public static void main(String[] args) throws InterruptedException {
        // Bounded queue with separate take/put locks
        BlockingQueue<String> pipeline = new LinkedBlockingQueue<>(10);

        // Producer Thread
        Thread producer = new Thread(() -> {
            try {
                for (int i = 1; i <= 5; i++) {
                    pipeline.put("Task-" + i); // Blocks if queue is full
                    System.out.println("Produced Task-" + i);
                    TimeUnit.MILLISECONDS.sleep(100);
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        // Consumer Thread
        Thread consumer = new Thread(() -> {
            try {
                while (true) {
                    String task = pipeline.take(); // Blocks if queue is empty
                    System.out.println("Consumed: " + task);
                    if (task.equals("Task-5")) break;
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        });

        producer.start();
        consumer.start();
        
        producer.join();
        consumer.join();
    }
}`} />

    Concurrent collections in Java minimize lock contention to scale multi-threaded applications. Using `ConcurrentHashMap` bucket locks for key-value storage, `CopyOnWriteArrayList` snapshots for read-heavy caches, and `LinkedBlockingQueue` dual-lock structures for pipelines are essential design patterns.

      <li><strong>Explain CopyOnWriteArrayList write performance:</strong> Writes copy the backing array under an exclusive lock, making writes expensive. It should only be used when reads vastly outnumber writes.</li>
      <li><strong>Contrast ArrayBlockingQueue vs LinkedBlockingQueue locks:</strong> ArrayBlockingQueue uses a single lock for both enqueue and dequeue, while LinkedBlockingQueue uses separate locks for takes and puts, allowing concurrent read/write operations.</li>
      <li><strong>What is a SynchronousQueue?</strong> A zero-capacity queue where each insert waits for a corresponding remove, facilitating direct thread handoffs.</li>
