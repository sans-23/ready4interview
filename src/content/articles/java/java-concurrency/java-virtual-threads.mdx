<p>
  For decades, Java threads operated on a 1:1 mapping model with Operating System (OS) kernel threads. While this provided access to OS multi-core schedulers, it limited scale: OS threads are heavy resources with fixed memory allocations and high context-switching costs. Project Loom (introduced in Java 21) decouples Java threads from OS threads via **Virtual Threads**.
</p>
<p>
  This guide compares platform threads with virtual threads, details the mechanics of carrier thread mounting and unmounting, explains lock pinning, and demonstrates how virtual threads simplify high-concurrency scaling.
</p>

<section className="content-section">
  <h2 id="loom-overview">Loom: Platform vs. Virtual Threads</h2>
  <p>
    Java divides threads into two categories:
  </p>

  <h3>1. Platform Threads (Legacy Model)</h3>
  <ul>
    <li>**1:1 OS Mapping:** Every `java.lang.Thread` instance is directly backed by an OS kernel thread.</li>
    <li>**High Resource Overhead:** Each platform thread allocates a fixed stack size, typically **1 Megabyte** of virtual memory. Running 10,000 threads consumes 10 GB of RAM, regardless of whether they are active or idle.</li>
    <li>**OS Context Switches:** When a platform thread blocks (e.g., waiting for database IO), the OS must switch thread contexts, saving registers and CPU states, which requires expensive transitions between user space and kernel space.</li>
  </ul>

  <h3>2. Virtual Threads (Project Loom Model)</h3>
  <ul>
    <li>**M:N Scheduling:** Many virtual threads are mapped to a small number of platform threads (**carrier threads**).</li>
    <li>**Heap Allocation:** Virtual thread stack frames are stored as standard objects in the **Java Heap** instead of kernel space. The initial footprint is minimal, typically starting at **a few hundred bytes** or kilobytes, and grows dynamically.</li>
    <li>**User-Space Scheduling:** Context switching is managed by the JVM in user space, avoiding OS-level interrupts. This enables scaling to millions of concurrent threads.</li>
  </ul>

<section className="content-section">
  <h2 id="mounting-mechanics">Mounting & Unmounting Mechanics</h2>
  <p>
    To execute, a virtual thread must be assigned to an underlying platform thread, which acts as its **carrier thread**. The JVM schedules virtual threads using a carrier pool backed by a `ForkJoinPool`.
  </p>

  <h3>1. Mounting</h3>
  <p>
    When a virtual thread becomes runnable, the JVM scheduler assigns it to a carrier thread. The scheduler copies the virtual thread's stack frames from the JVM heap to the carrier thread's actual execution stack.
  </p>

  <h3>2. Unmounting</h3>
  <p>
    When a virtual thread hits a blocking operation (such as thread sleep, lock acquisition, or socket read/write):
  </p>
  <ol>
    <li>The JVM intercepts the block, yields execution, and suspends the virtual thread.</li>
    <li>The virtual thread's stack frames are copied from the carrier thread back to the JVM heap.</li>
    <li>The carrier thread is unmounted, freeing it to execute other virtual threads in the queue.</li>
  </ol>
  <p>
    Once the blocking call completes, the scheduler re-queues the virtual thread. It will be mounted onto any available carrier thread (not necessarily the original one) to resume execution.
  </p>
</section>

<section className="content-section">
  <h2 id="lock-pinning">Pinning: Synchronized Block Risks</h2>
  <p>
    While virtual threads handle blocking operations efficiently, there are scenarios where a virtual thread cannot unmount from its carrier thread. This is known as **thread pinning**.
  </p>

  <h3>Why Pinning Occurs</h3>
  <p>
    A virtual thread becomes pinned to its carrier thread in two main cases:
  </p>
  <ol>
    <li>**Synchronized Blocks or Methods:** When a virtual thread executes code inside a `synchronized` block or method.</li>
    <li>**JNI Native Code:** When executing native code via Java Native Interface (JNI).</li>
  </ol>
  <p>
    If a virtual thread blocks while pinned (e.g. performing blocking database IO inside a synchronized block), the underlying **carrier thread is also blocked**. Since the carrier thread is a platform thread, blocking it defeats the purpose of virtual threads and can exhaust the carrier pool.
  </p>

  {/* Refactoring advice callout */}
  <Callout type="warning">
    <strong>Loom Migration Rule:</strong> When preparing code for virtual threads, refactor legacy `synchronized` blocks that contain blocking IO calls to use `ReentrantLock` instead. `ReentrantLock` is fully integrated with Loom's unmounting mechanics and does not cause pinning.
  </Callout>
</section>

<section className="content-section">
  <h2 id="scaling-models">Thread-Per-Request Scaling</h2>
  <p>
    Historically, web servers used two models to handle concurrent requests:
  </p>
  <ul>
    <li>**Thread-Pools:** Allocate one platform thread per request. This model is easy to write and debug but limits concurrency to the pool size (usually a few hundred).</li>
    <li>**Reactive Programming (RxJava/WebFlux):** Use non-blocking event loops. This scales well but requires complex callback architectures, splits stack traces, and makes debugging difficult.</li>
  </ul>
  <p>
    Project Loom offers the best of both worlds. Developers can write readable, synchronous, blocking code using the **thread-per-request** model, while virtual threads allow the application to scale to millions of concurrent requests under the hood.
  </p>
</section>

  <p>
    Virtual threads simplify high-concurrency Java scaling by decoupling threads from OS resources. By shifting stack allocations to the heap, virtual threads enable writing readable, blocking code that can scale to millions of concurrent tasks, provided developers avoid lock pinning issues.

      Key Interview Takeaways
    </h3>
    <ul>
      <li><strong>What is the core difference between platform and virtual threads?</strong> Platform threads map 1:1 to OS threads with a fixed 1MB memory footprint. Virtual threads are managed by the JVM, store stack frames on the heap, and start with a footprint of only a few hundred bytes, scaling to millions.</li>
      <li><strong>Explain carrier threads:</strong> Carrier threads are the platform threads used by the JVM's ForkJoinPool scheduler to run virtual threads.</li>
      <li><strong>What is thread pinning and how do you avoid it?</strong> Pinning occurs when a virtual thread cannot be unmounted from its carrier thread, typically during JNI calls or inside `synchronized` blocks. To avoid it, refactor synchronized blocks that contain blocking IO calls to use `ReentrantLock`.</li>
      <li><strong>Do virtual threads speed up CPU-bound tasks?</strong> No. Virtual threads scale IO-bound concurrency by freeing CPU resources during blocking operations. CPU-bound tasks require physical processors and will not see performance improvements from virtual threads.</li>
      <li><strong>Can virtual threads dead-lock?</strong> Yes. Virtual threads are `java.lang.Thread` instances and are subject to the same synchronization and dead-lock conditions as platform threads.</li>
    </ul>
