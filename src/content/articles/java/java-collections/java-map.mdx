<section className="content-section">
  <h2 id="hashmap-internals">HashMap Internals & Treeification</h2>
  <p>
    A Java <code>HashMap</code> represents a hash table structure. Let's analyze how it handles data allocation and collision mitigation:
  </p>

  <h3>1. Array of Buckets</h3>
  <p>
    Internally, HashMap stores values in a transient node array: <code>Node&lt;K,V&gt;[] table</code>. Each node implements <code>Map.Entry&lt;K,V&gt;</code> and contains:
  </p>
  <pre style={{ background: '#f5f5f5', padding: '12px', borderRadius: '6px', fontSize: '0.8rem', color: '#333', marginBottom: '16px' }}>
    int hash;<br/>
    final K key;<br/>
    V value;<br/>
    Node&lt;K,V&gt; next; // Linked list pointer for collisions
  </pre>

  <h3>2. Hash and Index Formulas</h3>
  <p>
    When you invoke <code>put(key, value)</code>, HashMap calculates a spread hash and maps it to a bucket:
  </p>
  <ul>
    <li><strong>Spreading:</strong> <code>hash = (h = key.hashCode()) ^ (h &gt;&gt;&gt; 16)</code>. Xoring the upper 16 bits to the lower 16 bits ensures that bit patterns affecting hash codes spread influence evenly across power-of-two table capacities.</li>
    <li><strong>Indexing:</strong> <code>index = (n - 1) & hash</code>, where $n$ is the current table array length. Since capacity is always a power of two ($2^x$), the bitwise <code>&</code> operates as a highly optimized modulo ($hash \pmod n$).</li>
  </ul>

  <h3>3. Collision Handling & Treeification</h3>
  <p>
    If different keys map to the same bucket index, they form a linked list. In Java 8+, if the linked list grows too long, the map optimizes retrieval via **Treeification**:
  </p>
  <ul>
    <li><strong>Treeify Threshold:</strong> If a bucket's list size reaches <code>TREEIFY_THRESHOLD = 8</code> AND the total table capacity is at least <code>MIN_TREEIFY_CAPACITY = 64</code>, the list nodes are converted into Red-Black tree nodes (<code>TreeNode</code>). This changes search complexity in high-collision buckets from $O(N)$ to $O(\log N)$.</li>
    <li><strong>Untreeify Threshold:</strong> During bucket resizing or removals, if a tree's node count falls to <code>UNTREEIFY_THRESHOLD = 6</code>, the node tree is converted back into a flat linked list to save memory overhead.</li>
  </ul>
  
  <Callout type="warning">
    <p>
      Treeification was introduced in Java 8 to prevent <strong>HashDoS</strong> security attacks, where attackers maliciously craft inputs that share identical bucket indexes (collisions) to overload a server's CPU. By converting linked lists to Red-Black trees, retrieval complexity remains bounded at $O(\log N)$ instead of degrading to $O(N)$.
    </p>
  </Callout>

  <h3>4. Load Factor & Resizing</h3>
  <p>
    The default threshold capacity is calculated as <code>capacity * loadFactor</code>. The default load factor is <code>0.75</code>. When the total number of mapping entries exceeds this threshold, the table undergoes **rehashing**:
  </p>
  <ul>
    <li>A new table is allocated with double the capacity: <code>newCapacity = oldCapacity &lt;&lt; 1</code>.</li>
    <li>All existing nodes are relocated. Since the new capacity is a power of 2, a node's index either stays the same or moves by exactly <code>oldCapacity</code>, allowing efficient bitwise re-allocation.</li>
  </ul>
</section>

<section className="content-section">
  <h2 id="treemap-internals">TreeMap: Sorted Tree Map</h2>
  <p>
    A <code>TreeMap</code> implements <code>NavigableMap</code> and is backed by a **Red-Black Tree**. It keeps keys sorted.
  </p>
  <p>
    <strong>Key TreeMap details:</strong>
  </p>
  <ul>
    <li><strong>Ordering:</strong> Kept in sorted order according to the keys' natural ordering (must implement the <code>Comparable</code> interface) or an explicit <code>Comparator</code> provided at instantiation.</li>
    <li><strong>Performance:</strong> Guarantees $O(\log N)$ time cost for operations like <code>containsKey</code>, <code>get</code>, <code>put</code>, and <code>remove</code>.</li>
    <li><strong>Null Keys:</strong> Does NOT allow <code>null</code> keys, because comparing a null key throws a <code>NullPointerException</code>. (Null values are allowed).</li>
    <li><strong>Range Operations:</strong> Provides utility methods like <code>subMap()</code>, <code>headMap()</code>, <code>firstKey()</code>, <code>lastKey()</code>, <code>floorKey()</code>, and <code>ceilingKey()</code>.</li>
  </ul>
</section>

<section className="content-section">
  <h2 id="linkedhashmap-internals">LinkedHashMap: Order-Preserving & Caching</h2>
  <p>
    A <code>LinkedHashMap</code> extends <code>HashMap</code>. While it uses the same hashing and bucket arrays, it wraps all node entries in a doubly-linked list structure:
  </p>
  <pre style={{ background: '#f5f5f5', padding: '12px', borderRadius: '6px', fontSize: '0.8rem', color: '#333', marginBottom: '16px' }}>
    static class Entry&lt;K,V&gt; extends HashMap.Node&lt;K,V&gt; &#123;<br/>
        Entry&lt;K,V&gt; before, after; // Pointers maintaining linked list order<br/>
    &#125;
  </pre>

  <h3>Access-Order vs. Insertion-Order</h3>
  <p>
    By default, LinkedHashMap maintains elements in **Insertion-Order**.
    However, you can configure it for **Access-Order** using the constructor:
    <br/>
    <code>new LinkedHashMap&lt;K,V&gt;(capacity, loadFactor, true)</code>
  </p>
  <p>
    When access-order is enabled, reading (<code>get</code>) or updating (<code>put</code>) an entry moves that node to the tail of the double-linked list. The element at the head represents the **Least Recently Used (LRU)** item.
  </p>

  <h3>Building an LRU Cache</h3>
  <p>
    LinkedHashMap includes a protected method: <code>protected boolean removeEldestEntry(Map.Entry&lt;K,V&gt; eldest)</code>. By default, it returns <code>false</code>. 
    If you override it to return <code>true</code> (e.g. when <code>size() &gt; maxCapacity</code>), the map automatically evicts the oldest (LRU) entry from the head whenever a new entry is added.
  </p>
</section>

public class JavaMapPlayground {
    public static void main(String[] args) {
        
        // 1. Basic Operations (HashMap)
        Map<String, Integer> map = new HashMap<>();
        map.put("Apple", 10);
        map.put("Banana", 20);
        map.put("Orange", 15);

        // Fetching with defaults
        int count1 = map.get("Apple"); // 10
        int count2 = map.getOrDefault("Grapes", 0); // 0 (prevents NullPointerException)

        // Verifications
        boolean hasApple = map.containsKey("Apple"); // true
        boolean hasValue20 = map.containsValue(20); // true

        // 2. Modern Java 8+ Functional APIs
        
        // computeIfAbsent: Computes value if key is missing, then inserts it
        map.computeIfAbsent("Mango", k -> 5 + 10); // Mango -> 15

        // computeIfPresent: Updates value if key exists
        map.computeIfPresent("Banana", (k, v) -> v + 10); // Banana -> 30

        // merge: Extremely useful for counters. Sums 5 to key "Orange".
        map.merge("Orange", 5, Integer::sum); // Orange -> 20 (15 + 5)
        map.merge("Watermelon", 1, Integer::sum); // Watermelon -> 1 (inserted)

        // putIfAbsent: Inserts only if absent, returns existing value if present
        map.putIfAbsent("Apple", 50); // Apple remains 10

        // 3. Map Iteration Mechanics
        
        // A. Entry Set Iterator (Most efficient)
        for (Map.Entry<String, Integer> entry : map.entrySet()) {
            System.out.println(entry.getKey() + " -> " + entry.getValue());
        }

        // B. Key Set Iteration (Avoid calling map.get() inside, causes lookup overhead)
        for (String key : map.keySet()) {
            System.out.println("Key: " + key);
        }

        // C. Lambda forEach
        map.forEach((k, v) -> System.out.println(k + ":" + v));

        // 4. Custom LRU Cache Implementation using LinkedHashMap
        LRUCache<String, String> cache = new LRUCache<>(3);
        cache.put("A", "Alpha");
        cache.put("B", "Beta");
        cache.put("C", "Gamma");
        
        cache.get("A"); // Accesses A, moves it to tail
        cache.put("D", "Delta"); // Exceeds capacity 3, evicts B (LRU)
        
        System.out.println("LRU Cache keys: " + cache.keySet()); // [C, A, D]
    }

    // Custom Thread-Safe or Non-Thread-Safe LRU Cache subclass
    public static class LRUCache<K, V> extends LinkedHashMap<K, V> {
        private final int maxCapacity;

        public LRUCache(int maxCapacity) {
            // true flag enables access-order tracking
            super(maxCapacity, 0.75f, true);
            this.maxCapacity = maxCapacity;
        }

        @Override
        protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
            // Automatically removes head element when capacity threshold is broken
            return size() > maxCapacity;
        }
    }
}`} />
