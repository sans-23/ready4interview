  <h2 id="priority-queue">PriorityQueue: Binary Heap Internals</h2>
  <p>
    A `PriorityQueue` is an unbounded queue based on a **priority heap** represented as a dynamically resizable array: `Object[] queue`. 
  </p>
  <p>
    <strong>Key PriorityQueue mechanics:</strong>
  </p>
  
  - **Heap Structure:** Implements a complete binary min-heap. For any element at array index `i`:
    - Its parent is located at index `(i - 1) / 2`.
    - Its left child is located at index `2i + 1`.
    - Its right child is located at index `2i + 2`.
  - **Ordering:** Elements are ordered according to their natural ordering (`Comparable`) or an explicit `Comparator` provided at instantiation.
  - **Null Entries:** Does NOT allow `null` elements, because comparing a null key throws a `NullPointerException`.

  <h3>Heap Adjustment Algorithms</h3>
  <p>
    To maintain the heap property (where the parent node is always smaller than or equal to its children), PriorityQueue executes sifting routines:
  </p>

  - **siftUp (Insertion):** When a new element is added via `offer()`, it is appended to the end of the array. The element is sifted up by repeatedly comparing it with its parent and swapping them if it violates the heap property. Complexity: $O(\log N)$.
  - **siftDown (Removal):** When the head (minimum element) is removed via `poll()`, the element at index 0 is extracted. The last element in the array is moved to index 0, and is sifted down by repeatedly swapping with its smallest child until the min-heap property is restored. Complexity: $O(\log N)$.
</section>

<section className="content-section">
  <h2 id="array-deque">ArrayDeque: Circular Array Buffer</h2>
  <p>
    An <code>ArrayDeque</code> is a resizable-array implementation of the <code>Deque</code> interface. It uses a **circular array buffer** and two pointers, <code>head</code> and <code>tail</code>, to track boundary indices.
  </p>
  <p>
    <strong>Key ArrayDeque mechanics:</strong>
  </p>
  - **No Node Allocation:** Unlike `LinkedList`, which allocates a new `Node` wrapper object for every insertion (increasing JVM memory footprint and Garbage Collection churn), `ArrayDeque` updates elements directly in its pre-allocated array.
  - **Circular Arithmetic:** The backing array capacity is always a power of two ($2^x$). This allows index increments and wrapping around boundary limits to use bitwise operations instead of division checks:
    <br/>
    `tail = (tail + 1) & (elements.length - 1)`
  - **Resizing:** When `head == tail`, the backing array is full. The array capacity doubles, and elements are copied to line up sequentially from index 0.
  - **Performance:** Offers amortized $O(1)$ cost for operations like `addFirst`, `addLast`, `pollFirst`, and `pollLast`.
</section>

<section className="content-section">
  <h2 id="caching-concurrency">Stack vs. ArrayDeque & BlockingQueue</h2>
  <p>
    A standard Java core interview question revolves around why legacy stack collection classes should be avoided:
  </p>

  <h3>1. java.util.Stack is Deprecated in Practice</h3>
  <p>
    The legacy <code>java.util.Stack</code> extends <code>Vector</code>. Because it is backed by Vector, every single operation (<code>push</code>, <code>pop</code>, <code>peek</code>) is **synchronized** using a synchronized lock.
  </p>
  <p>
    Even in a single-threaded environment, this introduces locking overhead, preventing JIT optimizations. In thread-safe contexts, it creates massive contention. 
    <br/>
    <em>Solution:</em> Always use <code>ArrayDeque</code> for LIFO stack operations:
    <br/>
    <code>Deque&lt;Integer&gt; stack = new ArrayDeque&lt;&gt;();</code>
  </p>

  <h3>2. Concurrency: BlockingQueues</h3>
  <p>
    For multi-threaded producer-consumer patterns, standard queues (like PriorityQueue and ArrayDeque) are unsafe. Instead, use implementations under <code>java.util.concurrent</code>:
  </p>
  - **ArrayBlockingQueue:** Bounded FIFO queue backed by a fixed-size array. Blocks producers when full and consumers when empty.
  - **LinkedBlockingQueue:** Optionally-bounded queue backed by linked nodes. Highly efficient for decoupled worker fleets.
  - **PriorityBlockingQueue:** Bounded priority queue that blocks consumers when empty, sorting tasks using binary heap locks.
  - **SynchronousQueue:** A zero-capacity handoff queue. Each insert must wait for a corresponding take, allowing zero buffering.
</section>

    public static void main(String[] args) {
        
        // 1. Standard FIFO Queue (LinkedList as backing)

        queue.offer("Alice");

        stack.push("Layer 1");
        stack.push("Layer 2");
