  const tabs = React.Children.toArray(children);

    navigator.clipboard.writeText(activeCode);

            background: 'none',
            border: '1px solid rgba(255, 255, 255, 0.1)',
            color: 'rgba(255, 255, 255, 0.5)',

<section className="content-section">
  <h2 id="array-list-internals">ArrayList: Growth & Copier Mechanics</h2>
  <p>
    An `ArrayList` is backed by a flat, contiguous object array: `transient Object[] elementData`. 
  </p>

  <h3>1. Resizing Mathematics</h3>
  <p>
    When elements are added and the backing array is full, the list grows dynamically. In Java, the capacity increases by exactly **50% (1.5x)** of the current size. The calculation uses bitwise right shifts for speed:
  </p>

```java
int oldCapacity = elementData.length;
int newCapacity = oldCapacity + (oldCapacity >> 1);
```

  <p>
    For example, if the current size is 10, `10 >> 1` equals 5. The new capacity becomes 15. Bit shifts are executed directly at the hardware processor register level, making them faster than floating-point math.
  </p>

  <h3>2. Dynamic Copying Costs</h3>
  <p>
    Once the new capacity is computed, `ArrayList` allocates a new array and copies the elements from the old array using `Arrays.copyOf(elementData, newCapacity)`. 
  </p>
  <p>
    This method internally calls the JVM's native `System.arraycopy` function. `System.arraycopy` utilizes highly optimized CPU memory block transfer instructions (such as `memmove`/`memcpy`), making it extremely fast, but it still has an $O(N)$ time complexity.
  </p>

  <h3>3. Index-Based Shifting</h3>
  <p>
    While appending an element to the end of an `ArrayList` is an amortized $O(1)$ operation, inserting or removing elements in the middle at index `k` requires shifting elements:
  </p>
  <ul>
    <li>All elements from index `k` to the end must be shifted to the right (for insertion) or left (for deletion).</li>
    <li>This requires copying elements, which is an $O(N)$ operation.</li>
  </ul>
</section>

<section className="content-section">
  <h2 id="linked-list-internals">LinkedList: Pointer Node & GC Overhead</h2>
  <p>
    `LinkedList` is backed by a doubly-linked structure of `Node` object wrappers. The class maintains references to `first` and `last`:
  </p>

```java
private static class Node<E> {
    E item;
    Node<E> next;
    Node<E> prev;
}
```

  <h3>1. Heavy Node Memory Footprint</h3>
  <p>
    Unlike `ArrayList`, which is just a single contiguous block of references, `LinkedList` allocates a new `Node` instance for every single element.
  </p>
  <p>
    On a typical 64-bit JVM with compressed OOPs (Ordinary Object Pointers):
  </p>
  <ul>
    <li>**Node Object Header:** 12 bytes + 4 bytes padding = 16 bytes.</li>
    <li>**References:** Three reference fields (`item`, `next`, `prev`) * 4 bytes each = 12 bytes.</li>
    <li>**Node Total:** 28 bytes, which rounds up to **32 bytes** due to JVM 8-byte object alignment rules.</li>
  </ul>
  <p>
    If you store 1,000,000 integers in a `LinkedList`, the wrapper nodes alone consume **32 megabytes** of heap space, in addition to the actual integer objects. In an `ArrayList`, storing 1,000,000 references requires only a single array of references (~4 megabytes).
  </p>

  <h3>2. Garbage Collection Pressure</h3>
  <p>
    Because each element requires a separate `Node` object allocation, adding and removing elements generates thousands of short-lived objects. This fills up the JVM's Young Generation area quickly, leading to frequent Minor Garbage Collection sweeps and potential CPU pauses.
  </p>

  <h3>3. Positional Access Traversal</h3>
  <p>
    To perform an index lookup `list.get(k)`, the list must be traversed sequentially from the closest end:
  </p>

```java
// Simplified traversal representation in LinkedList
Node<E> node = (index < (size >> 1)) ? first : last;
if (index < (size >> 1)) {
    for (int i = 0; i < index; i++) node = node.next;
} else {
    for (int i = size - 1; i > index; i--) node = node.prev;
}
```

  <p>
    Traversing elements in this way requires $O(N)$ steps, unlike the $O(1)$ instant index lookups in `ArrayList`.
  </p>
</section>

<section className="content-section">
  <h2 id="cache-locality">Hardware Alignment: CPU Cache Locality</h2>
  <p>
    Beyond asymptotic time complexity, the physical memory layouts of these structures impact hardware performance. Modern CPUs use L1, L2, and L3 caches to speed up data access.
  </p>

  <h3>Spatial Locality & Prefetching</h3>
  <ul>
    <li>
      <strong>ArrayList (Contiguous Memory):</strong> All elements in an array are stored in consecutive memory addresses. When the CPU accesses `elementData[i]`, the hardware prefetcher automatically loads the surrounding 64-byte chunk (a **cache line**) into L1/L2 cache. Subsequent reads of index `i+1`, `i+2` are instant cache hits.
    </li>
    <li>
      <strong>LinkedList (Scattered Memory):</strong> Since `Node` wrappers are allocated on the JVM heap at different times, they are scattered randomly across physical RAM. When the CPU dereferences `node.next`, it must fetch a completely different address, which often results in a **CPU Cache Miss**. The processor is forced to halt and wait for data to be loaded from main memory, which is significantly slower.
    </li>
  </ul>

  <div style={{ background: '#1a1b26', padding: '20px', borderRadius: '8px', border: '1px solid var(--border-color)', margin: '24px 0' }}>

    <p style={{ margin: 0, fontSize: '0.9rem', lineHeight: '1.6' }}>
      Due to cache locality, iterating through an `ArrayList` is often <strong>5x to 10x faster</strong> than traversing a `LinkedList` of the same size, even though both represent linear list loops.

        List<String> users = new CopyOnWriteArrayList<>();
        users.add("Alice");

        // Safe from ConcurrentModificationException during modification!

  <h2 id="conclusion">Conclusion & Takeaways</h2>
  <p>
    ArrayList is the default List implementation in Java due to its CPU cache-friendly layout and minimal memory overhead. Knowing when to bypass LinkedList's node creation overhead and how to use concurrent lists safely is essential for building scalable applications.
  </p>

      Key Interview Takeaways

    <ul>
      <li><strong>Which list is faster?</strong> ArrayList is almost always faster. Even for insertions in the middle, CPU cache-friendly contiguous copies outweigh LinkedList's node traversal steps in practice.</li>
      <li><strong>Explain ArrayList growth mechanics:</strong> ArrayList starts with a default size (usually 10) and increases capacity by 50% (`old + (old &gt;&gt; 1)`) when full. It uses `System.arraycopy` to transfer elements.</li>
      <li><strong>What is the memory cost of LinkedList?</strong> In 64-bit JVMs with compressed pointers, each element is wrapped in a `Node` object which consumes 32 bytes of heap space, in addition to the data itself, causing GC overhead.</li>
      <li><strong>When should you use LinkedList?</strong> Only when you need Deque operations at both ends and want to avoid array resizing overheads, though `ArrayDeque` is usually a better choice.</li>
      <li><strong>What is CopyOnWriteArrayList?</strong> A thread-safe list where write operations copy the backing array. Iteration is lock-free and immune to `ConcurrentModificationException`.</li>
    </ul>
